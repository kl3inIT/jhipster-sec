# Phase 8 - UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Angular + PrimeNG project; shadcn not applicable) |
| Preset | not applicable |
| Component library | PrimeNG 21.1.x on Aura with emerald primary. PrimeNG official components and patterns come first; custom wrappers or layout glue are allowed only when PrimeNG has no suitable primitive. Source: `frontend/package.json`, `frontend/src/app.config.ts`, user rule from this session |
| Icon library | PrimeIcons. Source: `frontend/package.json` |
| Font | Lato. Source: `frontend/src/assets/layout/_core.scss` |

PrimeNG-first rule:
- Prefer official PrimeNG components and patterns from `https://primeng.org/` for all buttons, tables, inputs, selectors, checkboxes, tags, inline messages, dialogs, toasts, and loading states.
- Custom UI is allowed only when PrimeNG does not have a suitable component for the use case. Custom code may provide route or state glue, layout composition, or thin wrappers around PrimeNG primitives, but it must not replace an available official PrimeNG component.
- Do not carry donor Bootstrap or ng-bootstrap UI primitives into Phase 8. Replace donor table, modal, badge, and form-control patterns with PrimeNG equivalents.

## Page And Interaction Contract

### `/admin/users` list

- Keep a dense admin table. Do not switch this phase to cards or a master-detail split list. Source: `08-CONTEXT.md` D-01.
- Column order is `login`, full name, `email`, activation status, assigned roles, last modified, and row actions. Source: `08-CONTEXT.md` D-02.
- Clicking the login value or the explicit view action opens `:login/view`. Edit stays a separate action. Source: `08-CONTEXT.md` D-03.
- Keep all primary row actions inline in the row. Do not move them into an overflow menu. Source: `08-CONTEXT.md` D-04.
- Put one prominent search field above the table. Search must match `login`, `email`, `firstName`, and `lastName`, debounce at `300ms`, reset paging to page 1 on change, and preserve normal paging and sort behavior. Source: `08-CONTEXT.md` D-07 to D-10; debounce duration is a default set in this spec.
- Use `20` rows per page with `[10, 20, 30]` page-size options to match existing entity list behavior. Source: existing `organization`, `department`, and `employee` list components.
- Use `p-table` loading state during fetch and refetch. Do not flash the empty state while a request is still in flight. Default set in this spec.
- The empty state lives inside the table `emptymessage` area, not outside the card.
- Use a non-interactive activation `p-tag` in the status column and keep the actual activate or deactivate action in the row actions cell.
- Use inline actions in this order: `View`, `Edit`, `Activate` or `Deactivate`, `Delete`. `View`, `Edit`, and `Delete` may be icon-only buttons; `Activate` or `Deactivate` keeps a visible text label because the verb must remain explicit. Source: `08-CONTEXT.md` D-15 to D-18 and phase defaults in this spec.
- Activation and deactivation run immediately with no confirmation dialog and return translated toast feedback. Source: `08-CONTEXT.md` D-15 and D-16.
- Delete stays inline but always opens a confirmation dialog with reject as the default focus target. Source: `08-CONTEXT.md` D-17 and current admin pattern in the menu definitions page.
- Self-deactivate and self-delete stay visible but disabled inline for the current admin row. Add accessible reason text in the button label or `aria-describedby`; do not rely on tooltip-only explanation. Source: `08-CONTEXT.md` D-18 and project accessibility rules.

### Detail and edit routes

- The primary detail and edit experience is route-based, not modal-based. Detail loads first, edit is an explicit follow-up action. Source: `08-CONTEXT.md` D-03.
- Use a split workspace: left card for user identity and audit data, right card for role assignment. Source: `08-CONTEXT.md` D-05 and D-11 to D-14.
- Detail mode is read-only. Enter edit only from an explicit `Edit` button. Source: `08-CONTEXT.md` D-06.
- Keep the role table visible on the detail route with disabled checkboxes so admins can inspect access before editing. Source: `08-CONTEXT.md` D-14.
- Do not move role assignment to a separate access page or secondary tab for this phase. Keep it on the same create and edit screen. Source: `08-CONTEXT.md` D-11.
- Show role code and human label or description in the right-side table. Source: `08-CONTEXT.md` D-13.
- Keep activation status visible in the left card header or identity summary, but do not make activation the primary form control. Activation changes belong to the list row action contract for this phase. Derived from `08-CONTEXT.md` D-15 and D-16.
- Use the same split layout for `new` and `edit`. For `new`, the left card starts empty and the right card still loads the authorities table for immediate assignment. Source: `08-CONTEXT.md` D-11 to D-13 and the preserved `/api/authorities` contract.
- `Back` and `Cancel` return to the prior list page, sort, and search state by reusing `WorkspaceContextService`. Source: `08-CONTEXT.md` canonical references and prior phase expectations.

### Loading, empty, and error states

- List blocking failure uses inline `p-message` inside the list card plus a retry button. Do not rely on toast alone for a page-blocking fetch failure. Default set in this spec.
- Mutation failures use translated `p-toast` and keep the current form or list state on screen.
- Detail or edit route loading uses two-card `p-skeleton` placeholders until both the user payload and the authorities payload are ready.
- Empty list and no-match search both use the same centered empty-state pattern inside the table, with copy that makes the next action obvious.

### Responsive contract

- Desktop breakpoint follows the existing shell behavior: `>= 992px` is desktop, `< 992px` is stacked mobile or tablet behavior. Source: `frontend/src/app/layout/service/layout.service.ts`.
- On desktop, use an `8/4` or visually equivalent left-right split between the user card and the role card.
- Below `992px`, stack the left card above the role card. Keep the search field full width and wrap toolbar actions onto a second line if needed.
- The user list stays a table at all widths. On narrower widths, use horizontal scroll rather than replacing the table with cards. Source: `08-CONTEXT.md` D-01 and D-02.

### Accessibility contract

- Every icon-only `p-button` must have a translated `aria-label`.
- The search control needs a visible label or an explicit `aria-label`; placeholder text alone is not enough.
- Associate the table with the page heading or a PrimeNG caption so screen readers announce context before column navigation.
- Each role checkbox must expose the role code and label together as its accessible name.
- After closing delete confirmation, return focus to the triggering row action. After save, land focus on the page heading or first blocking error.
- The screen must pass AXE and WCAG AA minimums, matching repo-level Angular requirements.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, tag-to-text spacing |
| sm | 8px | Compact row-action gaps, label-to-field spacing |
| md | 16px | Default control spacing, form rows, card body rhythm |
| lg | 24px | Toolbar spacing, card header-to-body separation |
| xl | 32px | Page section gaps, desktop split gutter |
| 2xl | 48px | Major section breaks and generous empty-state padding |
| 3xl | 64px | Page-level top or bottom breathing room |

Exceptions: `12px` table cell vertical padding in `p-datatable-sm` dense mode, `36px` square icon action buttons on desktop, and `44px` square touch targets for icon actions below `992px`. Defaults set in this spec.

## Typography

Only use sizes `14`, `20`, and `28` in this phase. Only use weights `400` and `600`. Defaults are aligned to the existing Lato base and the enterprise admin pattern already present in `frontend/`.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

## Color

Use current Aura theme tokens instead of hardcoding where possible. The values below are the visual contract the tokens should resolve to.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#f8fafc` via `var(--surface-ground)` | Page background and broad app surfaces |
| Secondary (30%) | `#ffffff` via `var(--surface-card)` | Cards, sidebar, topbar surface, table container background |
| Accent (10%) | `#10b981` via `var(--primary-color)` | `Create User`, `Save User`, active sort state, current page indicator, checked role checkboxes, activated status tag |
| Destructive | `#ef4444` | Delete action, destructive confirm action, destructive toast emphasis only |

Accent reserved for: `Create User`, `Save User`, the activated status tag, checked role-assignment checkboxes, the current paginator page, and the active sort affordance. Do not spend accent color on secondary navigation or neutral row actions.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Create User` |
| Empty state heading | `No users found` |
| Empty state body | `Try a different search or create a user to add the first account.` |
| Error state | `We couldn't load user data. Refresh the page or try again. If the problem continues, check that you still have admin access.` |
| Destructive confirmation | `Delete user`: `Delete user {{ login }}? This action cannot be undone.` |

Additional copy rules:
- Activation and deactivation do not use confirmation copy. They use immediate action plus translated toast feedback. Source: `08-CONTEXT.md` D-16.
- Self-deactivate and self-delete use disabled-state explanation text such as `You cannot deactivate your own account here.` Default set in this spec.
- Keep button labels verb-first and singular: `View`, `Edit`, `Activate`, `Deactivate`, `Delete`, `Save`, `Cancel`, `Back`.
