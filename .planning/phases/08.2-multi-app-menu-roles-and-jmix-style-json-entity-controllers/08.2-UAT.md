## Tests

### 1. Multi-App Role Menu Screen Loads Existing Grants
expected: Open the admin role security screen for a role that already has menu grants in at least two apps. In the Menu Access area, an app selector should be visible and should list both apps. The screen should not flatten everything into one tree, and the currently selected app should show only that app's menu structure and grants.
result: pass

### 2. Switching Apps Rebuilds The Menu Tree
expected: Change the app selector from one granted app to another. The visible menu tree should rebuild from backend menu definitions for the selected app, showing that app's structure and grant state without leaking menu ids or checkmarks from the previous app.
result: pass

### 3. First Grant Can Be Added In A Zero-Grant App
expected: Select an app that has menu definitions but no existing grants yet for the chosen role. That app should still appear in the selector. Grant one menu permission and save. The save should succeed, and reopening the role should show that new app grant persisted.
result: pass

### 4. Current-User Menu Resolution Stays App-Scoped
expected: For a user or role with different grants across apps, current-user navigation or `/api/security/menu-permissions?appName=...` should stay scoped to the requested app. Menus from app A should not bleed into app B.
result: pass

### 5. Secured Query Returns Only Allowed Rows And Fields
expected: Using a protected entity query endpoint such as `POST /api/organizations/query` with a simple top-level equality filter, the response should include only rows allowed by row policies. Allowed fields such as `code` and `name` should be present, while denied fields such as `budget` should stay hidden.
result: pass

### 6. Raw JSON Create Or Full Update Works On Protected Entities
expected: Using protected entity create or full update endpoints with raw JSON request bodies, saving an allowed organization, department, or employee should succeed without any map-wrapper contract. A follow-up read should reflect the allowed fields you changed.
result: pass

### 7. PATCH Preserves Omitted Fields
expected: Send a `PATCH` request that changes only one editable field on an allowed protected entity. Only that field should change, and omitted fields such as `code` should keep their previous value on the follow-up read.
result: pass

### 8. PATCH Still Blocks Denied Field Updates
expected: Try to `PATCH` a field the caller is not allowed to update, such as `budget`. The request should be rejected, and the denied field should remain unchanged afterwards.
result: pass
