# Phase 08.2: Multi-App Menu Roles and Jmix-Style JSON Entity Controllers - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend menu authorization so one role can carry app-scoped menu grants across multiple apps, and move secured entity REST boundaries from `Map<String, Object>` payload signatures toward Jmix-style raw JSON request/response flows. This phase must restore and preserve `loadByQuery` as a first-class secured capability and add explicit `PATCH` support for partial updates. It does not redesign auth/account/admin/mail APIs, it does not weaken CRUD, row-level, attribute-level, or fetch-plan enforcement, and it does not introduce database-backed fetch-plan definitions.

**Outcome**
- Roles can hold menu grants for multiple apps without losing app-scoped resolution.
- Protected entity controllers and services move to a raw JSON boundary for load/query/save flows instead of exposing `Map<String, Object>` request signatures.
- `loadByQuery` is restored end to end as a supported secured flow.
- `PATCH` exists with omitted-fields-untouched semantics and attribute-level write enforcement.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- **D-01:** Menu permissions remain app-scoped. The change is multi-app assignment per role, not global menu ids or cross-app leakage.
- **D-02:** Current-user navigation resolution remains per-app and backend-driven. Clients still resolve one app's menu visibility at a time.
- **D-03:** Admin flows must be able to view and manage a role's menu grants across multiple apps without duplicating role records or maintaining one role entity per app.
- **D-04:** Secured entity REST boundaries move toward Jmix-style raw JSON request/response payloads for create, update, query, and load flows instead of `Map<String, Object>` controller signatures.
- **D-05:** `loadByQuery` is in scope and must remain a first-class secured capability. Phase 08.2 is not complete if only list/loadOne/save/delete flows work.
- **D-06:** `PATCH` is in scope and must accept raw JSON request bodies, apply only provided fields, preserve omitted fields, and enforce attribute-level `EDIT` checks on every changed field.
- **D-07:** CRUD, row-level, attribute-level, and fetch-plan security remain locked protection targets across list, loadOne, loadByQuery, `PUT`, `PATCH`, and delete flows.
- **D-08:** Fetch plans stay defined in YAML or code builders only. This phase may refine how clients reference fetch plans over HTTP, but it must not introduce database-backed fetch-plan metadata.
- **D-09:** The Jmix docs demonstrate raw JSON entities/query bodies and partial-update semantics where omitted fields stay untouched; explicit HTTP `PATCH` is a deliberate project extension beyond the documented `PUT` examples. This is an inference from the Jmix docs plus the user requirement.

### Suggested Delivery Split
- **D-10:** Plan this phase as at least three substeps: multi-app menu permission modeling and admin APIs, raw JSON secured entity boundary plus `loadByQuery` restoration, and `PATCH` semantics plus regression and contract documentation.

### the agent's Discretion
- Exact raw JSON carrier type at the controller boundary (`JsonNode`, string-to-JSON parsing, or equivalent), as long as controller signatures stop exposing `Map<String, Object>`
- Exact admin API shape for cross-app role grant management
- Whether query-based loading is exposed as dedicated `/search` and `/query` endpoints or a consolidated JSON endpoint, as long as `loadByQuery` remains explicit and secured

</decisions>

<code_context>
## Existing Code Insights

### Current secured-entity boundary reality
- `OrganizationResource`, `DepartmentResource`, and `EmployeeResource` expose CRUD-only endpoints and accept `Map<String, Object>` request bodies for create and update. There is no `PATCH` endpoint and no HTTP query/search endpoint for secured entities today.
- `OrganizationService`, `DepartmentService`, and `EmployeeService` only call `loadList`, `loadOne`, `save`, and `delete`, so the `loadByQuery` contract is not reachable from the REST boundary.
- `SecureDataManager.loadByQuery(...)` still exists after Phase 08.1, but `SecureDataManagerImpl` currently throws when JPQL text is present and no web/API seam exercises the capability. That makes query-based loading effectively absent from clients.

### Current menu-permission reality
- `SecMenuPermission` storage is already app-scoped and unique by `(role, app_name, menu_id)`, which means multi-app support is partially present at the data-model level.
- `MenuPermissionResource` and `CurrentUserMenuPermissionService` resolve one `appName` at a time. That behavior is correct and must be preserved.
- `AdminMenuPermissionResource` currently creates one row at a time and queries a role's rows through a single-app filter, so role-centric multi-app management is not modeled cleanly yet.

### Current partial-update reality
- `PUT` update flows already pass only provided keys into `SecureMergeService.mergeForUpdate(...)`, which suggests omitted fields stay untouched in the merge layer, but there is no explicit `PATCH` contract or dedicated end-to-end coverage proving it.
- `SecureMergeService` and `SecureEntitySerializer` already sit in the secure path, so `PATCH` should build on the same attribute-level enforcement and serialization behavior instead of inventing a separate bypass path.

### Jmix alignment notes
- The Jmix load entities docs show raw JSON request and response bodies for list, search, and query flows, including fetch-plan-aware reads and query POST bodies.
- The Jmix update docs show partial-update semantics where omitted fields stay untouched and warn that collection updates replace what is present in the request unless the system can restore filtered items via `__securityToken`.
- The Jmix docs demonstrate partial updates via `PUT`, not `PATCH`; exposing explicit `PATCH` here is an intentional project-specific extension driven by the user requirement.

### Brownfield and security guardrails
- Phase 08.1 already pinned union-of-`ALLOW` semantics and preserved brownfield account/admin behavior. Phase 08.2 must not weaken those protections while extending the menu and entity-controller boundaries.
- Existing secured-entity integration coverage proves CRUD, row-policy filtering, and attribute-level write denial on `PUT`. Equivalent proof is missing for `loadByQuery` and `PATCH`, and nested collection or association behavior will need special care because row-level filtering can hide members from the client.

</code_context>

<risks>
## Risks And Guardrails

- **App-boundary leak risk:** Multi-app role assignment may accidentally flatten app scoping and cause one app's grants to appear in another app's navigation response.
- **Contract drift risk:** Replacing controller `Map<String, Object>` signatures with raw JSON can silently break validation, serialization shape, or existing frontend/admin consumers if the contract shift is not explicitly documented and regression-tested.
- **Query regression risk:** Restoring `loadByQuery` during the boundary refactor can reintroduce the Phase 08.1 regression or weaken row-level enforcement if query/search endpoints bypass the existing secure pipeline.
- **Patch semantics risk:** `PATCH` on associations or collections can cause unintended removals or security leaks if partial-update semantics diverge from the current secure merge behavior and row-filtered collection restore rules.
- **Over-alignment risk:** Trying to mirror every Jmix REST feature literally could violate project constraints around fetch-plan definition, brownfield safety, or the preserved `SecureDataManager` facade.

</risks>

<acceptance>
## Acceptance Criteria

- A single role can hold menu permissions for multiple apps in backend storage and admin APIs, while current-user menu resolution still returns only the requested app's allowed menu ids.
- Admin flows can inspect and manage menu grants for a role across apps without duplicating role records or abandoning app-scoped permission semantics.
- Protected entity controllers expose a Jmix-style raw JSON boundary for create, update, and query operations instead of `Map<String, Object>` request signatures.
- `loadByQuery` is restored and verified end to end as a first-class secured capability, including HTTP boundary coverage.
- `PATCH` accepts raw JSON request bodies, updates only provided fields, preserves omitted fields, and continues to enforce CRUD, row-level, attribute-level, and fetch-plan security.
- Existing list, loadOne, save, and delete flows keep their current CRUD, row-level, attribute-level, and fetch-plan security behavior.
- Brownfield auth/account/admin/mail flows and backend-driven navigation do not regress.
- Any intentional API contract changes are written down before implementation handoff.

</acceptance>
