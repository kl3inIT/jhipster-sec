## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Menu permissions remain app-scoped. The change is multi-app assignment per role, not global menu ids or cross-app leakage.
- **D-02:** Current-user navigation resolution remains per-app and backend-driven. Clients still resolve one app's menu visibility at a time.
- **D-03:** Admin flows must manage a role's menu grants across multiple apps without duplicating role records.
- **D-04:** Secured entity REST boundaries move away from `Map<String, Object>` controller signatures toward Jmix-style raw JSON request/response flows.
- **D-05:** `loadByQuery` must become a first-class secured capability at the HTTP boundary.
- **D-06:** `PATCH` must accept raw JSON, preserve omitted fields, and enforce attribute-level `EDIT` checks on changed fields.
- **D-07:** CRUD, row-level, attribute-level, and fetch-plan security stay locked protections across list, loadOne, loadByQuery, PUT, PATCH, and delete flows.
- **D-08:** Fetch plans remain YAML- or code-defined only.
- **D-09:** Jmix is the direction for raw JSON entity/query bodies and partial-update semantics, but explicit HTTP `PATCH` is a project-specific extension.

### Agent's Discretion

- Exact JSON carrier type at the controller and service seam (`JsonNode`, `ObjectNode`, thin request records, or an equivalent boundary type)
- Exact admin UX for editing one role's menu grants across multiple apps
- Exact secured query request shape, as long as the HTTP boundary becomes explicit, JSON-based, and brownfield-safe

### Deferred Ideas (OUT OF SCOPE)

- Database-backed fetch-plan definitions or fetch-plan authoring UI
- Arbitrary runtime query-language exposure beyond the protected query flow delivered in this phase
- Brownfield auth/account/admin/mail redesign outside the touched menu and secured-entity seams
- Phase 9 UX/performance polishing except where required to keep admin flows usable

## Summary

Phase 08.1 already repaired the internal secure data path, but Phase 08.2's outward boundary is still behind. `SecureDataManagerImpl` now has the right internal split, yet the public secured-entity API still exposes `Map<String, Object>` request bodies in `OrganizationResource`, `DepartmentResource`, and `EmployeeResource`, and those services only exercise list/loadOne/save/delete. `loadByQuery` exists in `SecureDataManagerImpl`, but there is no HTTP seam reaching it.

The admin menu area has the same pattern: the underlying storage model already supports multiple apps because `SecMenuPermission` is unique by `(role, app_name, menu_id)`, but the admin contract and frontend still behave as if menu grants are single-app. `AdminMenuPermissionResource` queries `findAllByRole(role)` and then filters one `appName` in memory, and the Angular `AdminMenuPermissionService` hardcodes `appName = 'jhipster-security-platform'`. Current-user navigation resolution is already correct and must stay unchanged.

The safest delivery sequence is additive rather than breaking:
1. Extend the admin menu-permission contract and frontend to model one role across many apps while leaving current-user `/api/security/menu-permissions?appName=...` untouched.
2. Move secured entity create/update/query request handling to raw JSON carriers at the web/service seam while keeping the existing on-wire CRUD response shapes stable for current frontend pages.
3. Add explicit secured query and `PATCH` endpoints, then close the phase with focused regression and contract documentation.

Two direct codebase findings shape the plan:
- The current frontend entity pages already send and receive plain JSON objects, so converting backend signatures from `Map<String, Object>` to raw JSON carriers is mostly wire-compatible for existing CRUD paths.
- All secured catalog entries currently set `jpqlAllowed(false)` in `MetamodelSecuredEntityCatalog`, and `SecureDataManagerImpl.loadByQuery(...)` throws on nonblank JPQL. That means Phase 08.2 should not promise arbitrary JPQL translation. The viable path is a Jmix-style JSON query body with a project-specific, explicitly limited query contract.

## Standard Stack

### Core

- Keep Java 25, Spring Boot 4, Spring MVC, Spring Data JPA, Jackson, and the existing `SecureDataManager` / `DataManager` / `UnconstrainedDataManager` stack.
- Keep `SecureMergeService`, `SecureEntitySerializer`, `FetchPlanResolver`, `RowLevelSpecificationBuilder`, and the secured entity catalog as the enforcement core.
- Keep the Angular frontend's existing `HttpClient` + typed interface pattern for current entity pages and role-security screens.

### Supporting

- No new runtime dependency is required for this phase.
- Jackson `JsonNode` / `ObjectNode` are already available and fit the raw JSON boundary well.
- Backend verification stays on JUnit 5 + Spring `@IntegrationTest` + `MockMvc`.
- Frontend verification stays on the existing Angular/Vitest test runner and current page/service specs.

### Alternatives Considered

- Full JPQL support in the HTTP query body now.
  Rejected: every current `SecuredEntityEntry` sets `jpqlAllowed(false)`, and `SecureDataManagerImpl` still throws for nonblank JPQL text.
- Replacing the existing GET list endpoint with POST query-only semantics immediately.
  Rejected: it would create avoidable brownfield risk for existing frontend entity pages that already consume `GET /api/{entities}` successfully.
- Keeping the admin menu query contract single-app and punting multi-app to a later phase.
  Rejected: it fails the core reason Phase 08.2 was inserted.

## Architecture Patterns

### Pattern 1: Preserve current-user menu resolution and widen only the admin contract

`CurrentUserMenuPermissionService` and `MenuPermissionResource` are already correct for runtime navigation:
- they resolve one `appName` at a time
- they use union-of-`ALLOW`
- they keep app boundaries intact

Do not change that behavior in this phase.

The admin path is where the change belongs:
- `AdminMenuPermissionResource` should support role-centric reads across all apps, with optional `appName` filtering rather than mandatory single-app filtering
- `SecMenuPermissionRepository` likely needs focused query helpers such as `findAllByRoleOrderByAppNameAscMenuIdAsc(...)` and/or `findAllByRoleAndAppName(...)`
- the Angular admin service and permission-matrix flow should stop assuming a single hardcoded app and instead surface the app dimension explicitly

This keeps runtime navigation stable while widening only the role-management view.

### Pattern 2: Treat raw JSON as a boundary adapter, not a secure-core rewrite

The secure core already works on `Map<String, Object>` and serialized maps. Phase 08.2 does not need to rewrite `SecureMergeService` or `SecureEntitySerializer` into a new internal representation.

The practical boundary pattern is:
- controllers accept raw JSON carriers (`JsonNode`, `ObjectNode`, or thin JSON-bound request records)
- application services convert those carriers into the existing secure-core input shape only at the boundary seam
- responses are returned as raw JSON trees or existing JSON object bodies derived from the secure serializer output

This isolates the refactor to the web/service boundary and keeps Phase 08.2 from ballooning into a secure-core redesign.

### Pattern 3: Make query support additive and explicitly limited

`SecureDataManagerImpl.loadByQuery(...)` already combines row-level constraints with the secure data path, but today it only supports the "no JPQL text" branch. Every catalog entry currently disables JPQL.

The planning-safe interpretation of "restore `loadByQuery`" is therefore:
- add explicit HTTP query endpoints such as `POST /api/organizations/query`, `POST /api/departments/query`, and `POST /api/employees/query`
- accept a JSON request body that carries pagination, sort, fetch-plan code, and a project-specific query shape
- keep JPQL either absent or explicitly rejected unless a later plan in this phase adds the missing catalog/config support and translation path

The key requirement is that secured query loading becomes reachable and tested end to end. It does not require full arbitrary query-language support.

### Pattern 4: Keep existing CRUD endpoints wire-compatible where possible

The current Angular entity services already send plain JSON bodies to:
- `POST /api/organizations`
- `PUT /api/organizations/{id}`
- equivalent department and employee endpoints

Those services are typed in TypeScript, but the wire payload is still ordinary JSON. That means the backend can move from `@RequestBody Map<String, Object>` to raw JSON carriers without forcing a broad frontend rewrite, as long as:
- endpoint paths stay stable for current CRUD flows
- response JSON fields remain unchanged
- validation and error semantics stay compatible

This is a good reason to make `PATCH` and secured query additive rather than destabilizing `GET` list/loadOne immediately.

### Pattern 5: Make `PATCH` use the existing secure merge path, not a second write pipeline

The current `save(...)` flow already merges only provided keys into the target entity through `SecureMergeService.mergeForUpdate(...)`. That is the right base for `PATCH`.

The phase should:
- expose explicit `@PatchMapping` endpoints on the secured entity resources
- parse raw JSON bodies into the same attribute map shape used by secure merge
- keep omitted fields absent from the merge input
- reuse the current secure serializer for the response

This preserves attribute-level `EDIT` enforcement and avoids inventing a parallel patch engine.

### Pattern 6: Separate menu work from entity boundary work into different plans

The codebase now has two independent risk clusters:
- multi-app admin menu-role assignment touches backend repository/resource logic plus the Angular permission matrix and service
- raw JSON query/PATCH touches three resources, three services, secure data request handling, and integration tests

They should not share a single execution plan. The natural split is:
- one plan for menu-role multi-app modeling and admin contract/UI updates
- one plan for secured entity raw JSON + query boundary work
- one plan for explicit PATCH semantics and regression/docs
- optionally one final regression/contract-note sweep if the planner decides the phase needs a fourth plan

### Anti-Patterns to Avoid

- Changing `MenuPermissionResource` or `CurrentUserMenuPermissionService` to return cross-app data
- Promising arbitrary JPQL support without changing `SecuredEntityEntry.jpqlAllowed` semantics and adding a real translation path
- Rewriting `SecureDataManager` internals again instead of adapting the web/service seam
- Making multi-app role assignment a pure backend feature without updating the Angular admin surface that already manages menu grants
- Introducing database-managed fetch plans or runtime fetch-plan authoring to support the new query endpoint

## Common Pitfalls

### Pitfall 1: Multi-app admin reads flatten app boundaries

**What goes wrong:** Admin role queries return one merged menu list without app grouping, and the Angular UI stops distinguishing which app a grant belongs to.

**Why it happens:** The current admin path already filters in memory by one `appName`, and the frontend service hardcodes one default app.

**How to avoid:** Keep `appName` on every admin DTO row, return rows grouped or sortable by app, and make the Angular service/API explicit about whether it is querying one app or all apps.

### Pitfall 2: Raw JSON boundary refactor accidentally changes existing CRUD wire shapes

**What goes wrong:** The controller signature change leaks into different field names, validation behavior, or response bodies, breaking current entity pages.

**Why it happens:** The code change looks internal, but existing Angular entity pages already rely on the current JSON field names and standard CRUD endpoints.

**How to avoid:** Keep current CRUD paths stable and treat raw JSON primarily as a server-side boundary refactor for create/update payload handling and future query/PATCH additions.

### Pitfall 3: `loadByQuery` gets exposed without a real query contract

**What goes wrong:** The phase adds a `/query` endpoint that simply wraps `SecuredLoadQuery.of(...)` and provides no actual query capability beyond the existing list endpoint.

**Why it happens:** The internal method name exists, so it is tempting to expose it without solving the missing request semantics.

**How to avoid:** Make the plan spell out an explicit request body and how that body becomes a secured specification or equivalent filter input.

### Pitfall 4: The plan assumes JPQL support that the catalog forbids

**What goes wrong:** Tasks reference JPQL translation or Jmix query text, but every current `SecuredEntityEntry` still sets `jpqlAllowed(false)` and `SecureDataManagerImpl` throws on nonblank JPQL.

**Why it happens:** The Jmix docs show query bodies, which can be misread as "must support arbitrary JPQL right now."

**How to avoid:** Keep the phase query contract project-specific and explicit. If JPQL is needed later, plan it as a separate follow-up with catalog/config and translation work.

### Pitfall 5: `PATCH` bypasses attribute security

**What goes wrong:** A new patch endpoint updates only changed keys, but it shortcuts the secure merge path and skips per-attribute `EDIT` checks.

**Why it happens:** Patch work is often implemented as a special-case update flow.

**How to avoid:** Route `PATCH` through the same secure merge and save path as `PUT`, differing only in which keys are present in the parsed JSON payload.

### Pitfall 6: Multi-app menu assignment is planned as backend-only

**What goes wrong:** The backend supports multiple apps per role, but the Angular permission matrix still loads and saves only `'jhipster-security-platform'`.

**Why it happens:** The existing frontend already works, so it can be mistaken for out-of-scope.

**How to avoid:** Treat the Angular `AdminMenuPermissionService` and `permission-matrix.component.*` as part of the same contract surface as the backend admin resource.
