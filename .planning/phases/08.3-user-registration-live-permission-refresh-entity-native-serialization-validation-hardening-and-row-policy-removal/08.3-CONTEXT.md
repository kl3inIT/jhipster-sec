# Phase 08.3: User Registration, Live Permission Refresh, Entity-Native Serialization, Validation Hardening, And Row Policy Removal - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 08.3 repairs the remaining security and secured-data gaps that were left after Phase 08.2. The backend already exposes `/api/register`, but the standalone frontend still has no registration flow. Current-user permissions are still effectively frozen at login because JWT and frontend caches keep the original authority snapshot. The secured entity path still depends on `String`, `Map<String, Object>`, and `JsonNode` carriers through core security seams. JSON-based secured controllers still need explicit fail-closed validation. Row policy is still present even though the product direction is now to remove it entirely.

This phase adds frontend self-registration on top of the preserved backend contract, refreshes authorities from database state without forcing logout or login, moves the secured entity flow toward real typed entities, hardens JSON validation, and removes row policy from schema, backend, frontend, and tests.

**Outcome**
- `frontend/` exposes a real registration route and form backed by the preserved `/api/register` contract.
- Current-user permission checks, menu visibility, and secured-entity capabilities observe updated authorities without re-authentication.
- Secured entity internals operate on real entity instances, with JSON parsing and serialization isolated to explicit edge adapters.
- JSON-based secured controllers validate malformed bodies, unknown fields, invalid references, and unsafe query shapes explicitly.
- Row policy is removed completely from runtime code, schema, admin APIs, frontend UI, and tests.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- **D-01:** Frontend user registration is in scope. The preserved backend `/api/register` flow stays the primary contract unless code review proves a brownfield blocker that must be fixed safely.
- **D-02:** Permission changes must take effect without logout or login. JWT remains the identity carrier, but current request authorities must come from database state rather than the login-time snapshot.
- **D-03:** An `AcceptsGrantedAuthorities`-style mechanism or equivalent request-time principal replacement is required so refreshed authorities can be injected into the current authenticated user representation.
- **D-04:** Secured entity core flow must operate on real entity instances. `String`, `Map<String, Object>`, and `JsonNode` are allowed only at transport or adapter edges, not as the core security data representation.
- **D-05:** Validation for JSON-based secured controllers must be explicit and fail closed. DTO-only boundaries are not required if strict binding and validation are implemented correctly.
- **D-06:** Row policy is removed completely from the product in this phase, including schema, backend providers, admin APIs, frontend UI, tests, and supporting enums.
- **D-07:** Phase 9 depends on this phase. UX and performance work should not proceed on top of stale authorities, map-based secure flow, or row-policy debt.

### Suggested Delivery Split
- **D-08:** Plan this phase as five workstreams: registration parity, live authority refresh, typed secured-data flow, validation hardening, and row-policy retirement.
- **D-09:** Treat row-policy removal as the last wave after live-authority refresh, typed secured flow, and validation hardening have replacement coverage.

### Agent's Discretion
- Exact request-time authority refresh hook in Spring Security, as long as the current authenticated user sees DB authorities without re-login
- Exact typed secure adapter shape between HTTP JSON and real entities
- Exact validation implementation using Jackson configuration, explicit binders, and `jakarta.validation.Validator`
- Whether registration needs small backend boundary cleanup in addition to the missing frontend flow

</decisions>

<code_context>
## Existing Code Insights

### Registration reality
- `/api/register` already exists in `AccountResource` and uses `ManagedUserVM`.
- `UserService.registerUser(...)` lowercases login and email, creates activation state, and always assigns `ROLE_USER` regardless of incoming authorities.
- Backend registration already has integration coverage.
- The missing piece is the standalone Angular registration route, page, service, and public navigation wiring.

### Live-authority reality
- `AuthenticateController` puts an `auth` claim into JWT based on `authentication.getAuthorities()` at login time.
- `DomainUserDetailsService.UserWithId` is currently immutable from a granted-authority perspective.
- `MergedSecurityContextBridge` still reads authorities from the current `Authentication`, so if the request authentication is stale, downstream permission checks stay stale too.
- `frontend/` also caches current-user account, menu permissions, and secured-entity capabilities until login state changes.

### Typed secured-data reality
- Protected entity controllers accept `String` request bodies, parse them into `JsonNode`, and serialize responses back into raw JSON strings.
- `OrganizationService`, `DepartmentService`, and `EmployeeService` still adapt JSON trees into `Map<String, Object>` and call map-based `SecureDataManager` methods.
- `DataManager` and `UnconstrainedDataManager` are already typed internally, which means the remaining map-based debt is concentrated in the secure facade, serializer, merge, and entity service or resource seams.

### Validation reality
- Current JSON controllers explicitly reject only non-object JSON in their local parse helpers.
- `SecuredEntityQueryVM` has little or no explicit bean validation.
- There is no shared explicit binder or validator for secured JSON payloads, so malformed shape, unknown fields, invalid nested references, and unsafe query structure can still slip through too far.

### Row-policy reality
- Row-policy schema, runtime classes, admin APIs, frontend screens, and tests still exist.
- Some secured-entity integration tests still rely on row-policy behavior for ownership-style filtering.
- Row-policy removal will therefore require both code deletion and replacement of any remaining test assumptions that still depend on it.

</code_context>

<risks>
## Risks And Guardrails

- **Auth refresh risk:** Request-time authority refresh can break security if identity and authorities drift apart or if only some current-user surfaces use the refreshed principal.
- **Scope-creep risk:** A typed secured-entity rewrite can expand uncontrollably if it tries to redesign every public contract instead of replacing internal core carriers first.
- **Validation regression risk:** Enforcing unknown-field and invalid-reference rejection can surface brownfield callers that accidentally depended on permissive parsing.
- **Removal-order risk:** Removing row policy before typed secured flow and validation are stable can create temporary security regressions and unclear failing tests.
- **Frontend staleness risk:** Backend live-refresh work is insufficient if `frontend/` keeps stale account, menu, or capability caches after the user remains logged in.

</risks>

<acceptance>
## Acceptance Criteria

- `frontend/` exposes a working user registration flow on top of the preserved `/api/register` backend behavior.
- Updating a user's authorities in persistent storage changes current-user permission outcomes, menu visibility, and secured-entity capabilities without requiring logout or login.
- The secured entity data path operates on real typed entities internally and no longer depends on `Map<String, Object>`, raw `String`, or `JsonNode` as the core security representation.
- JSON-based secured controllers reject malformed, unsafe, or invalid payloads explicitly instead of relying on incidental behavior.
- Row-policy code, schema, admin UI, and tests are removed completely, and no surviving logic depends on row-policy providers or enums.
- Brownfield auth, account, admin, and mail flows remain green after the refactor.

</acceptance>

<deferred>
## Deferred Ideas

- Phase 9 UX or performance work except where a regression or cache invalidation change is required by live permission refresh
- Database-backed fetch-plan definitions or runtime fetch-plan authoring
- Broad auth or account API redesign unrelated to preserved registration behavior
- A future replacement for row policy; this phase only removes the current implementation

</deferred>
