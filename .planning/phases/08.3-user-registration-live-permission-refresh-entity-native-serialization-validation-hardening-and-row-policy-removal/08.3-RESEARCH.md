## Summary

Phase 08.3 is not a speculative cleanup. The codebase already shows five concrete gaps:

1. `/api/register` exists and is tested in the backend, but the standalone Angular frontend still has no registration route or page.
2. JWT and frontend caches keep a login-time authority snapshot, so role changes do not take effect until re-authentication.
3. The secured entity pipeline still moves through raw strings, `JsonNode`, and `Map<String, Object>` even though the internal `DataManager` and `UnconstrainedDataManager` are already typed.
4. JSON-based secured controllers only do narrow shape checks today and still need explicit binder or validator behavior for unknown fields and invalid references.
5. Row policy remains present in schema, runtime code, admin APIs, frontend, and tests even though the product direction is now to remove it.

The safest delivery sequence is:
1. Add frontend registration on top of the already-preserved backend contract.
2. Refresh authorities at request time and invalidate frontend permission caches.
3. Move secured entity internals to typed entity flow with explicit JSON edge adapters.
4. Harden validation on top of the typed path.
5. Remove row policy only after the replacement permission model is stable.

Two codebase findings are especially important:
- The backend registration flow is already brownfield-safe, which means the main product gap is frontend parity rather than a new backend design.
- `DataManager` and `UnconstrainedDataManager` are already typed, so the remaining map or string debt is localized enough to plan a real refactor rather than another compatibility wrapper.

## Standard Stack

### Core

- Keep Java 25, Spring Boot 4, Spring Security, Spring MVC, Jackson, Hibernate Validator, and the existing `DataManager` or `UnconstrainedDataManager` infrastructure.
- Keep the standalone Angular frontend structure, PrimeNG Sakai shell, current account or auth services, and focused Angular or Vitest specs.
- Keep JWT as the authentication token format, but stop trusting the JWT authority claim as the only source of current request authorities.

### Supporting

- No new runtime dependency is required for this phase.
- `jakarta.validation.Validator` and existing bean-validation infrastructure are already available for explicit payload checks.
- Jackson binding is already present and can support strict JSON edge adapters instead of manual string parsing.
- Backend verification should continue to use Gradle `integrationTest`, not `test`, because the repository excludes `*IT` from the default test task.

### Alternatives Considered

- Keep frontend registration out of scope and rely on admin user creation only.
  Rejected: the backend register flow already exists, and the user explicitly requested standalone registration.
- Reuse JWT `auth` claims as the authoritative source and just clear frontend caches.
  Rejected: backend method-security and current-user permission checks would still be stale.
- Leave `SecureDataManager` map-based internally and only add more adapters.
  Rejected: it preserves the exact type and validation drift that caused this phase.
- Keep row policy and just hide the UI.
  Rejected: the user asked for complete removal, and the complexity remains in schema and runtime if only the UI is removed.

## Architecture Patterns

### Pattern 1: Preserve backend registration, add frontend parity

`AccountResource.registerAccount(...)` and `UserService.registerUser(...)` already define the contract:
- input still uses `ManagedUserVM`
- persisted users still receive `ROLE_USER`
- activation behavior and related mail flow remain preserved

Phase 08.3 should treat this as a stable backend contract and add the missing frontend route, form, service, and validation around it. Backend changes should only happen if current integration tests reveal a real brownfield gap.

### Pattern 2: Treat JWT as identity carrier, not authority source of truth

The current stale-authority problem exists because login-time authorities are embedded in the token and then reused across the session. The safer pattern is:
- keep the token as user identity and session continuity
- resolve current authorities from the database for each request or for each authenticated principal refresh
- replace or mutate the authenticated principal so downstream current-user helpers, method security, and services all see the refreshed authorities

This fits the user's `AcceptsGrantedAuthorities` direction without requiring session state.

### Pattern 3: Refresh frontend permission state from one authority-aware source

Even after backend request-time refresh lands, `frontend/` still caches:
- current account state
- menu permissions
- secured-entity capabilities

The plan should give those caches one invalidation or refresh path tied to current-user authority changes, rather than leaving each feature to refresh independently.

### Pattern 4: Use typed entity flow internally and keep JSON only at edges

The codebase already has typed internal managers. The remaining refactor target is the secured facade and entity services:
- HTTP request JSON binds into typed entities or typed command objects at the edge
- security checks and secured data operations run on real entities internally
- HTTP responses serialize from real entities through an explicit serializer at the edge

This is the practical path to the user's target shape for `UnconstrainedDataManager` without destabilizing every public endpoint at once.

### Pattern 5: Put explicit validation next to JSON binding

Validation hardening should be attached to the JSON edge adapter rather than spread across controller ad hoc checks:
- reject non-object payloads immediately
- reject unknown fields explicitly
- validate query payload structure before secure query execution
- validate references and bean constraints before merge or save

This avoids a DTO-only rewrite while still making validation fail closed.

### Pattern 6: Remove row policy after the replacement path is proven

Row policy touches schema, provider logic, enums, admin APIs, frontend UI, and tests. The safest order is:
- first land live-authority refresh
- then land typed secured flow
- then land validation hardening
- only then delete row-policy code and update tests that depended on it

This keeps the system from losing two security mechanisms at once while the secured path is still being refactored.

### Anti-Patterns to Avoid

- Treating the JWT `auth` claim as the final authority source after Phase 08.3
- Rewriting every auth or account boundary when the real gap is frontend parity plus authority refresh
- Keeping map-based secure internals and only renaming interfaces to look typed
- Enforcing stricter JSON validation in controllers while services still accept raw unvalidated maps
- Removing row-policy code before secured-entity tests are converted away from row-policy assumptions

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Frontend registration | Entirely new account contract | Existing `/api/register` backend behavior plus `angapp` donor flow | Preserves brownfield backend safety |
| Live authority refresh | Session-only mutable auth store | Request-time principal refresh with DB authorities | Works with stateless JWT and current request security |
| Typed secured flow | Another compatibility layer of `Map<String, Object>` wrappers | Typed entity operations plus explicit JSON edge adapters | Removes the core type drift instead of hiding it |
| Validation hardening | More ad hoc `if` checks in each controller | Shared strict binder or validator service plus bean validation | Centralizes fail-closed behavior |
| Row-policy retirement | Hide the UI and keep runtime code | Delete schema, runtime, UI, and tests together | Avoids dead feature paths and hidden coupling |

## Common Pitfalls

### Pitfall 1: Backend refresh lands, frontend still looks stale

**What goes wrong:** Menu and capability decisions remain stale because Angular caches do not refresh even though the backend now resolves authorities from DB.

**How to avoid:** Put cache invalidation and refresh into the same plan as backend live-authority refresh.

### Pitfall 2: Typed refactor stops at controller signatures

**What goes wrong:** Controllers stop accepting strings, but services and the secure facade still convert back into raw maps immediately.

**How to avoid:** Make the typed flow reach the secure merge, serializer, and entity-service seams, not just the controller method signature.

### Pitfall 3: Validation hardening becomes a DTO rewrite

**What goes wrong:** The phase turns into a wide DTO migration that is not required for correctness.

**How to avoid:** Focus on strict JSON binding and explicit validation behavior, regardless of whether a DTO is present.

### Pitfall 4: Row-policy deletion strands tests

**What goes wrong:** Tests that still depend on row-policy ownership behavior start failing with no replacement proof.

**How to avoid:** Convert or replace those tests as part of the removal plan instead of treating deletion as a pure cleanup task.

### Pitfall 5: Live authority refresh updates only one current-user surface

**What goes wrong:** `/api/account` shows fresh authorities, but menu or capability endpoints still evaluate stale permissions.

**How to avoid:** Prove the same token sees changed authorities across account, menu, and secured-entity capability endpoints.
