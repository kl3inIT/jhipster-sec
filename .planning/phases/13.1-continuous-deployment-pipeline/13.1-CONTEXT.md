# Phase 13.1: Continuous Deployment Pipeline - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a CD workflow that publishes the Docker image to GitHub Container Registry (GHCR) and then SSH-deploys to the production server after backend CI passes on main. This phase covers the new `deploy.yml` GitHub Actions workflow only — no changes to the existing CI lanes from Phase 13, no new test infrastructure.

</domain>

<decisions>
## Implementation Decisions

### Registry
- **D-01:** Publish to GitHub Container Registry (`ghcr.io/<owner>/<repo>`) using `GITHUB_TOKEN` — no extra secrets needed for registry auth.
- **D-02:** Do NOT use Docker Hub. GHCR is the correct choice for a private GitHub-hosted project: no pull rate limits from CI runners, zero extra credentials, free private storage.

### CD Trigger
- **D-03:** The deploy workflow triggers via `workflow_run` on `Backend CI` completion (type: `completed`) scoped to `branches: [main]`. The deploy job must check `github.event.workflow_run.conclusion == 'success'` before proceeding — failed CI must not trigger deployment.
- **D-04:** This keeps CI and CD in separate workflow files. Do not add a deploy job inside `backend.yml`.

### Image Tagging
- **D-05:** Push two tags on each successful deployment: `:latest` (convenience) and `:sha-<short_sha>` (reproducibility). Short SHA from `github.event.workflow_run.head_sha` (first 7 chars).

### Build and Publish Step
- **D-06:** Build the Jib image and push directly to GHCR using `./gradlew -Pprod jib` (Jib's `jib` task — builds and pushes to remote registry in one step, no local Docker daemon required).
- **D-07:** Jib `to.image` must be configured as `ghcr.io/<owner>/<repo>:latest` with credentials injected via Jib environment variables or Gradle properties from `GITHUB_TOKEN`.

### SSH Deploy
- **D-08:** After the image push succeeds, SSH into the production server and run: `docker compose pull && docker compose up -d`. Use the existing `src/main/docker/app.yml` compose manifest.
- **D-09:** SSH credentials stored as GitHub Actions repo secrets: `DEPLOY_SSH_KEY` (private key) and `DEPLOY_HOST` (server address). Use `webfactory/ssh-agent` action to load the key, or inline `ssh-keyscan` + `ssh` command.
- **D-10:** The deploy user on the server must have permission to run `docker compose` commands and access to the `app.yml` file path. This is a setup concern, not a workflow file concern — document the required server-side setup in the plan.

### Post-Deploy Verification
- **D-11:** After `docker compose up -d`, poll `http://<server>/management/health/readiness` until HTTP 200 or 60-second timeout. If timeout fires, the job fails — no silent broken deployments.
- **D-12:** Use `DEPLOY_HOST` secret to construct the health check URL dynamically (e.g., `http://${{ secrets.DEPLOY_HOST }}/management/health/readiness`).

### Workflow File
- **D-13:** New file: `.github/workflows/deploy.yml`. Named `Continuous Deployment` in the workflow `name` field.
- **D-14:** No changes to `backend.yml`, `frontend.yml`, or `prod-validation.yml` from Phase 13.

### Claude's Discretion
- Exact Jib credential injection approach (environment variable vs Gradle `-P` flag)
- Whether to use `appleboy/ssh-action` or raw `ssh` command for the deploy step
- Exact `docker compose` file path assumption (~/app.yml vs hardcoded path — document in plan)
- Timeout values for health check polling (60s suggested)

</decisions>

<specifics>
## Specific Ideas

- The server is behind VPN / Nginx Proxy Manager — the health check URL may need to use the internal address rather than the public domain. The plan should note that `DEPLOY_HOST` secret value and the health check endpoint must be confirmed for the actual network topology.
- Use `workflow_run` trigger (not `push`) to ensure CD never runs when CI is broken. This is the correct GitHub Actions pattern for "deploy after CI passes."

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `backend.yml` workflow name is `Backend CI` — the `workflow_run` trigger must reference this exact string.
- `src/main/docker/app.yml` is the full-stack Compose manifest; the deploy SSH step must reference this file on the production server.
- Jib conventions in `buildSrc/src/main/groovy/jhipster.docker-conventions.gradle` define the image name as `jhipster-sec:latest`; the deploy workflow overrides the target registry to GHCR.

### Established Patterns
- Phase 13 workflows all use `ubuntu-latest` runners and `actions/checkout@v4` + `actions/setup-java@v4` + `gradle/actions/setup-gradle@v4` — deploy workflow follows the same baseline.
- `GITHUB_TOKEN` is automatically available in all GitHub Actions workflows; no extra secret setup for GHCR authentication.

### Integration Points
- `.github/workflows/deploy.yml` — new file; triggered by `Backend CI` workflow completion.
- `DEPLOY_SSH_KEY` and `DEPLOY_HOST` — new GitHub repo secrets to be configured by the user before the workflow can run.
- Server-side: production server must have the `app.yml` file accessible and `docker compose` available for the deploy user.

</code_context>
