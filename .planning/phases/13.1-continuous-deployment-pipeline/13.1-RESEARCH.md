# Phase 13.1: Continuous Deployment Pipeline - Research

**Researched:** 2026-04-06
**Domain:** GitHub Actions CD -- Jib/GHCR image push + SSH remote deploy
**Confidence:** HIGH

## Summary

Phase 13.1 adds a single new `.github/workflows/deploy.yml` file. The workflow triggers via `workflow_run` on `Backend CI` completion, builds and pushes the application image to GHCR using Jib's `jib` task, then SSH-deploys to the production server with `docker compose pull && docker compose up -d`, and finishes by polling the Spring Boot readiness probe.

All design decisions are locked in CONTEXT.md. The research focuses on: exact Jib GHCR credential injection, `workflow_run` trigger syntax, SSH action tradeoffs, health-check polling pattern, required secrets, and ASVS L1 security considerations. No new test infrastructure is required for this phase -- the workflow itself is the deliverable.

**Primary recommendation:** Inject Jib GHCR credentials via `-Djib.to.auth.username` / `-Djib.to.auth.password` system properties on the Gradle command line (no Docker daemon needed), with `packages: write` job permission for `GITHUB_TOKEN`. Use `appleboy/ssh-action` for the deploy step for its built-in fingerprint verification and clean error surface. Use a `bash` polling loop for the health check -- no third-party action needed.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Publish to GitHub Container Registry (`ghcr.io/<owner>/<repo>`) using `GITHUB_TOKEN` -- no extra secrets needed for registry auth.
- **D-02:** Do NOT use Docker Hub. GHCR is the correct choice for a private GitHub-hosted project.
- **D-03:** Deploy workflow triggers via `workflow_run` on `Backend CI` completion (type: `completed`) scoped to `branches: [main]`. The deploy job must check `github.event.workflow_run.conclusion == 'success'`.
- **D-04:** CI and CD stay in separate workflow files. No deploy job inside `backend.yml`.
- **D-05:** Push two tags on each successful deployment: `:latest` and `:sha-<short_sha>` (first 7 chars of `github.event.workflow_run.head_sha`).
- **D-06:** Build and push using `./gradlew -Pprod jib` (no local Docker daemon required).
- **D-07:** Jib `to.image` must be `ghcr.io/<owner>/<repo>:latest` with credentials injected via Jib environment variables or Gradle properties from `GITHUB_TOKEN`.
- **D-08:** After image push, SSH into production server and run `docker compose pull && docker compose up -d` using `src/main/docker/app.yml`.
- **D-09:** SSH credentials stored as `DEPLOY_SSH_KEY` (private key) and `DEPLOY_HOST` (server address) GitHub repo secrets. Use `webfactory/ssh-agent` or inline `ssh-keyscan` + `ssh`.
- **D-10:** Deploy user on the server must have `docker compose` access -- document server-side setup, not a workflow concern.
- **D-11:** After `docker compose up -d`, poll `http://<server>/management/health/readiness` until HTTP 200 or 60-second timeout. Job fails on timeout.
- **D-12:** Use `DEPLOY_HOST` secret to construct health check URL dynamically.
- **D-13:** New file: `.github/workflows/deploy.yml`. Workflow `name` field: `Continuous Deployment`.
- **D-14:** No changes to `backend.yml`, `frontend.yml`, or `prod-validation.yml`.

## Architecture Patterns

### Workflow File Location

```
.github/
  workflows/
    backend.yml          # Backend CI (existing -- do not touch)
    frontend.yml         # Frontend CI (existing -- do not touch)
    prod-validation.yml  # Prod validation (existing -- do not touch)
    deploy.yml           # NEW -- Continuous Deployment (this phase only)
```

### Pattern 1: workflow_run Trigger with Conclusion Guard

**What:** Trigger a workflow when a named workflow completes, guard the deploy job with a conclusion check so failed CI never deploys.

**When to use:** CD must only run after CI passes on the main branch.

```yaml
# Source: https://docs.github.com/en/actions/writing-workflows/choosing-when-your-workflow-runs/events-that-trigger-workflows#workflow_run
# [CITED: docs.github.com]
on:
  workflow_run:
    workflows: ["Backend CI"]   # MUST match name: field in backend.yml exactly
    types: [completed]
    branches: [main]

jobs:
  deploy:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
```

**Critical gotcha:** The `deploy.yml` file itself MUST be on the default branch (main) for `workflow_run` to activate. During initial rollout, merge `deploy.yml` to main before expecting it to fire.

**Workflow name match:** The string `"Backend CI"` must exactly match the `name:` field at the top of `backend.yml`. Current value confirmed: `name: Backend CI`. [VERIFIED: .github/workflows/backend.yml]

### Pattern 2: Jib Push to GHCR via Gradle CLI

**What:** Override the Jib `to.image` and inject credentials at the Gradle command line so the remote registry receives the image without a local Docker daemon.

**When to use:** CI runners without Docker daemon (or where daemon setup is wasteful).

```yaml
# Source: https://github.com/GoogleContainerTools/jib/tree/master/jib-gradle-plugin#authentication-methods
# [CITED: GoogleContainerTools/jib docs]
- name: Build and push Docker image
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  run: |
    SHORT_SHA=$(echo "${{ github.event.workflow_run.head_sha }}" | cut -c1-7)
    ./gradlew -Pprod jib \
      -Djib.to.image=ghcr.io/kl3init/jhipster-sec:latest \
      -Djib.to.tags=sha-${SHORT_SHA} \
      -Djib.to.auth.username=${{ github.actor }} \
      -Djib.to.auth.password=${GITHUB_TOKEN}
```

**Notes:**
- `-Djib.to.tags=sha-${SHORT_SHA}` adds the second tag alongside `:latest` (Jib pushes all configured tags in a single operation). [VERIFIED: Jib docs]
- `jib.to.auth.username` = `${{ github.actor }}` (the GitHub username running the workflow). [CITED: docs.github.com/publishing-docker-images]
- `jib.to.auth.password` = `GITHUB_TOKEN` (automatically scoped to the repository's packages). [CITED: docs.github.com]
- The `GITHUB_TOKEN` can push to GHCR only if the job has `packages: write` permission.

**Required job-level permissions:**
```yaml
permissions:
  contents: read
  packages: write
```
[CITED: docs.github.com/publishing-packages/publishing-docker-images]

**Image name for this project:** `ghcr.io/kl3init/jhipster-sec` [VERIFIED: git remote = https://github.com/kl3inIT/jhipster-sec.git, GHCR normalizes owner to lowercase]

### Pattern 3: SSH Deploy with appleboy/ssh-action

**What:** Run remote commands on the production server using a stored private key secret.

**When to use:** Single production server, simple deploy commands, want built-in fingerprint/error handling.

```yaml
# Source: https://github.com/appleboy/ssh-action
# [CITED: appleboy/ssh-action README]
- name: Deploy to production
  uses: appleboy/ssh-action@v1
  with:
    host: ${{ secrets.DEPLOY_HOST }}
    username: ${{ secrets.DEPLOY_USER }}
    key: ${{ secrets.DEPLOY_SSH_KEY }}
    script: |
      cd ~/app
      docker compose -f app.yml pull
      docker compose -f app.yml up -d
```

**Discrete variables needed:**
- `DEPLOY_HOST` -- hostname or IP of the production server (no port in the secret; use `port` input if non-22)
- `DEPLOY_USER` -- OS username on the server with Docker access (e.g., `deploy` or `ubuntu`)
- `DEPLOY_SSH_KEY` -- full unencrypted private key PEM (ed25519 or RSA)

### Pattern 4: Health-Check Polling with bash

**What:** Poll the Spring Boot readiness actuator endpoint until HTTP 200 or timeout, then fail the job.

**When to use:** Post-deploy verification without third-party action.

```yaml
# Source: Adapted from prod-validation.yml pattern in this project
# [VERIFIED: .github/workflows/prod-validation.yml]
- name: Verify deployment health
  run: |
    echo "Polling readiness at http://${{ secrets.DEPLOY_HOST }}/management/health/readiness ..."
    timeout 60 bash -c \
      'until curl -sf http://${{ secrets.DEPLOY_HOST }}/management/health/readiness; \
       do sleep 3; done'
    echo "Deployment is healthy."
```

**Notes:**
- `timeout 60` causes a non-zero exit after 60 seconds, which fails the job. [ASSUMED -- standard bash behavior, well-known]
- `curl -sf` returns non-zero for HTTP errors (suppress output noise). [ASSUMED]
- 3-second poll interval gives up to ~20 attempts within the 60-second window.
- The existing `prod-validation.yml` uses `timeout 120 bash -c 'until curl -sf ...; do sleep 2; done'` -- this phase uses 60s as a tighter production SLA signal.
- **Network topology note:** The production server is behind VPN and Nginx Proxy Manager. The health check URL must be reachable from the GitHub Actions runner. If `DEPLOY_HOST` is an internal IP, the runner will not reach it and health checks will always timeout. The plan must document that `DEPLOY_HOST` must be the publicly reachable hostname (the Nginx Proxy Manager ingress domain or public IP), and the `/management/health/readiness` path must be routed through Nginx to port 8080 of the app container.

### Anti-Patterns to Avoid

- **Deploying on CI failure:** Never omit the `if: github.event.workflow_run.conclusion == 'success'` guard. Without it, a failed CI will still trigger the deploy job.
- **Hardcoding image tags in build.gradle `to.image`:** The Jib convention currently outputs `jhipster-sec:latest` (local). The deploy workflow must override `jib.to.image` on the CLI -- do not commit GHCR credentials or image paths into the build convention file.
- **Using `jibDockerBuild` in CI:** `jibDockerBuild` requires a local Docker daemon. The remote push variant is `jib` (no `Docker` suffix). `prod-validation.yml` uses `jibDockerBuild` because it runs the container locally on the runner -- the deploy workflow must use `jib` instead.
- **Omitting `packages: write` permission:** Without this, `GITHUB_TOKEN` cannot push to GHCR and Jib auth will fail silently with a 401.
- **Referencing a non-existent workflow name:** If `workflows: ["Backend CI"]` does not exactly match the `name:` in `backend.yml`, the trigger never fires.

## Common Pitfalls

### Pitfall 1: workflow_run only fires from the default branch

**What goes wrong:** `deploy.yml` is created on a feature branch. Pushes to main trigger `backend.yml` but `deploy.yml` never fires because the workflow file itself is not on the default branch.

**Why it happens:** GitHub evaluates `workflow_run` listeners from the repository's default branch (main). A workflow file that only exists on a non-default branch is invisible to the event system.

**How to avoid:** Merge `deploy.yml` to main before testing. Confirm the workflow appears under the `Actions` tab of the repository.

**Warning signs:** No deploy run appears in the Actions tab after a successful backend CI on main.

### Pitfall 2: GHCR image visibility / package not linked to repository

**What goes wrong:** The first push creates a new GHCR package that is `private` by default but is not yet linked to the repository. Subsequent pulls by the production server (using `GITHUB_TOKEN` or a PAT) may fail.

**Why it happens:** New GHCR packages require a one-time manual step in the GitHub UI to link them to the repository and set visibility.

**How to avoid:** After the first successful push, navigate to `https://github.com/users/kl3inIT/packages` (or the org equivalent), find the `jhipster-sec` package, and connect it to the `jhipster-sec` repository. The production server's docker pull will then authenticate via its own PAT or the `GITHUB_TOKEN` of a server-side credential.

**Warning signs:** `docker compose pull` on the server returns `unauthorized` or `manifest unknown`.

### Pitfall 3: Production server cannot pull from GHCR

**What goes wrong:** `docker compose pull` on the server fails because the server is not authenticated to `ghcr.io`.

**Why it happens:** `GITHUB_TOKEN` is a workflow-scoped token that exists only inside the GitHub Actions runner. The production server needs its own credentials to pull from a private GHCR package.

**How to avoid:** On the production server, run `docker login ghcr.io -u <github-username> --password-stdin` with a classic PAT that has `read:packages` scope. Store the credential in `~/.docker/config.json` for the deploy user. This is a one-time server setup step.

**Warning signs:** `docker compose pull` exits non-zero with an auth error during the SSH deploy step.

### Pitfall 4: DEPLOY_HOST health check URL unreachable from GitHub runner

**What goes wrong:** The health check step times out immediately even though the app started correctly.

**Why it happens:** The production server is behind VPN and Nginx Proxy Manager. `DEPLOY_HOST` set to an internal/VPN-only IP is not reachable from GitHub's hosted runners.

**How to avoid:** `DEPLOY_HOST` must be the publicly accessible domain routed through Nginx Proxy Manager, and Nginx must proxy `/management/health/readiness` to port 8080 of the app container. If the app is not publicly exposed, a VPN-connected runner or a self-hosted runner on the LAN is required.

**Warning signs:** `curl: (28) Connection timed out` in the health-check step. App is healthy on the server but the workflow job fails.

### Pitfall 5: Jib builds stale source (no checkout of head_sha)

**What goes wrong:** `deploy.yml` checks out the default branch tip instead of the specific commit that triggered the backend CI run. The built image may not match what CI validated.

**Why it happens:** `actions/checkout@v4` defaults to `github.sha` (the SHA that triggered the current workflow), but `workflow_run` events set `github.sha` to the head of the default branch, not the triggering run's SHA.

**How to avoid:** Use `ref: ${{ github.event.workflow_run.head_sha }}` in the checkout step:
```yaml
- uses: actions/checkout@v4
  with:
    ref: ${{ github.event.workflow_run.head_sha }}
```

**Warning signs:** Subtle drift between what CI tested and what gets deployed; harder to detect than an outright failure.

### Pitfall 6: `jibDockerBuild` vs `jib` task confusion

**What goes wrong:** Copying the `prod-validation.yml` Gradle command (`jibDockerBuild`) into the deploy workflow causes a failure because there is no Docker daemon on the runner in the deploy context.

**Why it happens:** `jibDockerBuild` pushes to the local Docker daemon's image store. The deploy workflow must use `jib` (the daemon-less remote push task).

**How to avoid:** Always use `./gradlew -Pprod jib` (not `jibDockerBuild`) in the deploy workflow.

**Warning signs:** `Cannot connect to Docker daemon` error during the image build step.

## Code Examples

### Full deploy.yml skeleton

```yaml
# Source: All patterns verified from official GitHub Actions docs and Jib docs
# [CITED: docs.github.com, GoogleContainerTools/jib]
name: Continuous Deployment

on:
  workflow_run:
    workflows: ["Backend CI"]
    types: [completed]
    branches: [main]

permissions:
  contents: read
  packages: write

jobs:
  deploy:
    name: Build, Push, Deploy
    runs-on: ubuntu-latest
    timeout-minutes: 20
    if: ${{ github.event.workflow_run.conclusion == 'success' }}

    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.workflow_run.head_sha }}

      - name: Set up JDK 25
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: 25

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: npm

      - name: Set up Gradle
        uses: gradle/actions/setup-gradle@v4

      - name: Install root npm dependencies
        run: npm ci

      - name: Build and push Docker image to GHCR
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          SHORT_SHA=$(echo "${{ github.event.workflow_run.head_sha }}" | cut -c1-7)
          ./gradlew -Pprod jib \
            -Djib.to.image=ghcr.io/kl3init/jhipster-sec:latest \
            -Djib.to.tags=sha-${SHORT_SHA} \
            -Djib.to.auth.username=${{ github.actor }} \
            -Djib.to.auth.password=${GITHUB_TOKEN}

      - name: Deploy to production server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd ~/app
            docker compose -f app.yml pull
            docker compose -f app.yml up -d

      - name: Verify deployment health
        run: |
          echo "Polling readiness at http://${{ secrets.DEPLOY_HOST }}/management/health/readiness ..."
          timeout 60 bash -c \
            'until curl -sf "http://${{ secrets.DEPLOY_HOST }}/management/health/readiness"; \
             do sleep 3; done'
          echo "Deployment is healthy."
```

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|------------------|
| V2 Authentication | yes (GHCR + SSH) | `GITHUB_TOKEN` for registry; ed25519 private key for SSH |
| V3 Session Management | no | Stateless workflow; no session state |
| V4 Access Control | yes | Least-privilege job permissions (`packages: write`, `contents: read`) |
| V5 Input Validation | no | No user input; all values from trusted GHA context |
| V6 Cryptography | yes | SSH key strength; do not use RSA-1024 or DSA |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|--------------------|
| Secret leakage in workflow logs | Information Disclosure | Never echo secrets directly; GHA masks `secrets.*` values automatically |
| Compromised SSH private key | Elevation of Privilege | Use ed25519 keys; rotate on suspected compromise; restrict `authorized_keys` to deploy commands only |
| Malicious PR triggering deploy | Elevation of Privilege | `workflow_run` + `branches: [main]` + `conclusion == 'success'` guard: only merges to main that pass CI trigger the deploy |
| Overly broad `GITHUB_TOKEN` permissions | Elevation of Privilege | Declare minimal `permissions:` block at job level: `contents: read`, `packages: write` |
| Docker socket abuse on server | Elevation of Privilege | Deploy user should be in `docker` group but NOT have unrestricted `sudo`. Deploy commands are limited to `docker compose pull + up -d`. |
| GHCR image poisoning | Tampering | Image is built from the exact `head_sha` of the commit that passed CI; SHA-tagged images provide audit trail |

## Planning Resolutions

1. **Health check URL network reachability**
   - Resolution: Treat this as an operator prerequisite, not an implementation unknown. The plan requires `DEPLOY_HOST` to be a GitHub-runner-reachable hostname or ingress address that serves `http://<host>/management/health/readiness` through the actual VPN / Nginx Proxy Manager topology.
   - Planning impact: The workflow will use `http://${{ secrets.DEPLOY_HOST }}/management/health/readiness` exactly, and the plan will document that a self-hosted runner is required if the endpoint is not publicly reachable.

2. **app.yml path on the production server**
   - Resolution: Use `~/app/app.yml` as the concrete default server-side compose path for planning and execution, and document it as a prerequisite the operator must satisfy before first deploy.
   - Planning impact: The deploy step will run `docker compose -f ~/app/app.yml pull` and `docker compose -f ~/app/app.yml up -d`, with the path assumption called out explicitly in the workflow text.

3. **GHCR username choice in workflow_run context**
   - Resolution: Use `github.repository_owner` instead of `github.actor` for `jib.to.auth.username`.
   - Planning impact: This avoids collaborator-trigger edge cases and keeps the GHCR target aligned with the repository owner path.
