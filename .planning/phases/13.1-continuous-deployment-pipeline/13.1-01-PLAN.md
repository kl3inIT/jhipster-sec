<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → GHCR | The workflow exchanges package-publish credentials and pushes production images to the remote registry. |
| GitHub Actions runner → production server over SSH | The workflow crosses from hosted CI infrastructure into the production host using repository secrets and a verified host key. |
| production server → GHCR | The remote host pulls a private image package during deployment and must already have pull credentials configured. |
| public or ingress health endpoint → GitHub Actions runner | Post-deploy verification depends on the runner reaching the readiness endpoint through the actual network topology. |

## Execution Gate

- Any unresolved **high** severity threat blocks execution of this plan.
- Task verification must prove SSH host authenticity setup (`ssh-keyscan -H ... >> ~/.ssh/known_hosts`), least-privilege workflow permissions, and fail-closed readiness behavior before the plan can pass.
- Secrets may only flow through `secrets.*`, the SSH agent input, or masked environment variables; no task may print secret values.

## STRIDE Threat Register

| Threat ID | Category | Severity | Component | Disposition | Mitigation Plan |
|-----------|----------|----------|-----------|-------------|-----------------|
| T-13.1-01 | E | medium | `.github/workflows/deploy.yml` job permissions | mitigate | Declare only `contents: read` and `packages: write`; do not grant broader repository or admin permissions. |
| T-13.1-02 | T | high | Jib image source selection | mitigate | Checkout `github.event.workflow_run.head_sha` and build that exact commit so the deployed image matches the CI-validated source. |
| T-13.1-03 | E | high | workflow trigger path | mitigate | Use `workflow_run` for `Backend CI` on `branches: [main]` plus `github.event.workflow_run.conclusion == 'success'` so failed CI and non-main activity cannot deploy. |
| T-13.1-04 | I | high | SSH and registry secrets in workflow logs | mitigate | Pass secrets only through `secrets.*` inputs or env values, never echo them, and keep deploy commands free of debug output that prints credentials. |
| T-13.1-05 | S | high | SSH host authenticity | mitigate | Require `ssh-keyscan -H ${{ secrets.DEPLOY_HOST }}` to populate `known_hosts` before the raw `ssh` command so the runner verifies the remote host identity. |
| T-13.1-06 | D | medium | deployment health verification | mitigate | Poll `/management/health/readiness` with a hard 60-second timeout and fail the job on timeout so broken deployments are surfaced immediately. |
| T-13.1-07 | T | medium | remote image pull | mitigate | Document the required server-side `docker login ghcr.io` setup and use the existing `app.yml` compose manifest so the pull path is explicit and auditable. |
| T-13.1-08 | D | medium | DEPLOY_HOST reachability through VPN / ingress topology | accept | The workflow cannot solve network topology itself; document that `DEPLOY_HOST` must be runner-reachable or a self-hosted runner is required. |
</threat_model>
