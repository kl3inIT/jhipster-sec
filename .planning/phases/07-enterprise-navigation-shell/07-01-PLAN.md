Create a new Liquibase changelog `src/main/resources/config/liquibase/changelog/20260325000100_create_sec_navigation_grant.xml` and wire it into `src/main/resources/config/liquibase/master.xml`.

The new table must be named `sec_navigation_grant` and must contain:
- `id` as bigint primary key using the standard sequence generator pattern already used by security entities
- `authority_name` varchar(50) not null
- `app_name` varchar(100) not null
- `node_id` varchar(150) not null
- `effect` varchar(10) not null

Add a unique constraint on `(authority_name, app_name, node_id)` so one authority cannot store both ALLOW and DENY for the same node id in the same app.
Add indexes on `(app_name, authority_name)` and `(app_name, node_id)` to support current-user lookups.

Create `src/main/java/com/vn/core/security/domain/SecNavigationGrant.java` as a JPA entity under the security domain with fields `id`, `authorityName`, `appName`, `nodeId`, and `effect`.
Model `effect` as a string field constrained to the stored values `ALLOW` or `DENY`; do not expand `TargetType` and do not reuse `SecPermission`.

Create `src/main/java/com/vn/core/security/repository/SecNavigationGrantRepository.java` with:
- `List<SecNavigationGrant> findAllByAppNameAndAuthorityNameIn(String appName, Collection<String> authorityNames)`
- `List<SecNavigationGrant> findAllByAppName(String appName)`

Do not add any admin-write REST UI in this plan. Phase 7 only needs current-user read access for menu visibility and route gating.

Create `src/main/java/com/vn/core/service/dto/security/NavigationGrantResponseDTO.java` with fields `String appName` and `List<String> allowedNodeIds`.

Create `src/main/java/com/vn/core/service/security/CurrentUserNavigationGrantService.java` that:
- injects `MergedSecurityService` and `SecNavigationGrantRepository`
- exposes `List<String> getAllowedNodeIds(String appName)`
- returns an empty list when the current user has no authorities
- loads all grants for the current user's authorities and requested app
- applies deny-wins semantics so any `DENY` for a node id excludes it even when another authority allows it
- sorts the returned node ids alphabetically for a stable REST contract

Create `src/main/java/com/vn/core/web/rest/NavigationGrantResource.java` under `web/rest` with:
- `@RestController`
- `@RequestMapping("/api/security")`
- `@PreAuthorize("isAuthenticated()")`
- `@GetMapping("/navigation-grants")`
- a required `@RequestParam("appName") String appName`
- a `ResponseEntity<NavigationGrantResponseDTO>` response

Add backend tests:
- `CurrentUserNavigationGrantServiceTest` covering no-authority empty result, multiple-authority allow union, and deny-wins conflict resolution for `jhipster-security-platform`
- `NavigationGrantResourceIT` covering authenticated success for `/api/security/navigation-grants?appName=jhipster-security-platform`, unauthenticated rejection, and the exact JSON fields `appName` and `allowedNodeIds`

Keep the endpoint read-only. Do not create POST, PUT, or DELETE handlers in this plan.
