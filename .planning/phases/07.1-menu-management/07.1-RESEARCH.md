## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Menu permission assignment is added to the existing role screen as a "Menu Access" tab or section — role-first approach, consistent with the established permission matrix pattern.
- **D-02:** Menu nodes are displayed in a tree/hierarchy view matching the navigation registry structure (grouped by section: Home, Entities, Security), not a flat list.
- **D-03:** `SecMenuDefinition` supports full admin CRUD — admins can create, edit, and delete menu node entries, not just view them.
- **D-04:** The primary use case for admin-created nodes is dynamic menu items: admin registers ad-hoc entries (e.g. external links, custom dashboards) that the frontend renders at runtime from the backend catalog.
- **D-05:** The frontend renders dynamic nodes via backend-driven rendering — it fetches the full menu catalog from the backend at runtime and merges dynamically-created nodes alongside the static registry entries. The backend becomes the source of truth for dynamic entries.
- **D-06:** Menu definition management lives at a dedicated page: `/admin/security/menu-definitions`. List of all `SecMenuDefinition` entries with Create/Edit/Delete actions. Separate from the permission assignment screen.
- **D-07:** No `SecMenuPermission` row = **deny**. Admins must explicitly grant each menu node per role. Secure-by-default, consistent with Phase 7 deny-wins resolution.
- **D-08:** A new `AdminMenuPermissionResource` provides CRUD for `SecMenuPermission` (assign/revoke per role+menu). Secured to `ROLE_ADMIN`.
- **D-09:** A sync endpoint seeds/refreshes `SecMenuDefinition` from the frontend navigation registry. Admins can trigger this to add newly-registered frontend nodes to the catalog.
- **D-10:** Existing `MenuPermissionResource` (current-user GET) is not modified — admin endpoints live in a separate resource.

## Summary

Phase 07.1 delivers two collaborating features: (1) a standalone admin CRUD page for `SecMenuDefinition` catalog entries at `/admin/security/menu-definitions`, and (2) a "Menu Access" tab added to the existing role permissions screen at `/admin/security/roles/:name/permissions`. Both are backed by new admin REST endpoints in `com.vn.core.web.rest.admin.security`.

The domain entities `SecMenuDefinition` and `SecMenuPermission` already exist with their full JPA mappings and repositories in place. The Liquibase migrations have been applied. The frontend navigation registry (`APP_NAVIGATION_TREE`) that provides the tree structure is also in place. This means the phase is entirely about wiring the admin UI and backend service/controller layer around already-existing domain objects — no schema work is required.

The pattern is well-established in the project: all three deliverables (backend resource, frontend service, frontend list+dialog component) have direct analogues in the Roles and Row Policies features. The Permission Matrix extension for "Menu Access" follows the same pending-change + flush pattern as the existing entity permission tab, with a tree component replacing the entity selector panel.

**Primary recommendation:** Build the backend admin resource and DTO layer first (Wave 1), then the menu definitions frontend page (Wave 2), then extend the permission matrix with the Menu Access tab (Wave 3). The sync endpoint can ship alongside the backend resource in Wave 1.

### Pattern 2: Sync Endpoint (POST trigger design — Claude's discretion)

```
POST /api/admin/sec/menu-definitions/sync
Body: { "appName": "jhipster-security-platform" }
Action: for each node in APP_NAVIGATION_TREE (flatmap to leaves):
  - if findByAppNameAndMenuId returns empty → insert new SecMenuDefinition
  - if already exists → skip (idempotent, do not modify)
Response: { "seeded": N, "skipped": M }
```

The frontend navigation registry is the source of truth. The backend sync endpoint receives node data from the request body (the frontend sends it) because the backend has no native access to the Angular registry at runtime.

Alternative: frontend sends the full registry snapshot in the POST body; backend upserts only missing entries. This is the simplest design and avoids a startup hook that would create ordering dependencies.

### Anti-Patterns to Avoid

- **Modifying MenuPermissionResource:** D-10 locks this. The current-user GET endpoint must not be touched.
- **Storing effect = 'GRANT' from admin UI:** `SecMenuPermission.effect` uses `ALLOW|DENY` at the storage layer (not the `GRANT` string used by `SecPermission`). The admin endpoint for menu permissions stores and reads `ALLOW` directly — no `SecPermissionUiContractService` normalization needed.
- **Fetching tree structure from backend:** The tree structure for the Menu Access tab comes from importing `APP_NAVIGATION_TREE` directly into the Angular component. Only the granted node IDs come from the API.
- **ChangeDetectionStrategy.Default on new components:** All new Angular components must use `ChangeDetectionStrategy.OnPush` per project convention.
- **`standalone: true` in decorator:** Per CLAUDE.md, this must NOT be set — it is the default in Angular v20+.
- **Using `@HostBinding`/`@HostListener`:** Forbidden per CLAUDE.md; use `host` object in decorator instead.
- **Using `@Input()` / `@Output()` decorators:** Use `input()` and `output()` functions instead per CLAUDE.md. However, existing dialog components use `@Input`/`@Output` — match the existing dialog pattern for consistency (RoleDialogComponent and RowPolicyDialogComponent use `@Input`/`@Output`).

## Common Pitfalls

### Pitfall 1: SecMenuPermission.effect stores ALLOW/DENY, not GRANT
**What goes wrong:** Frontend sends `effect: 'GRANT'` to the admin menu permission endpoint (copying the SecPermission pattern), but `SecMenuPermission` uses `ALLOW|DENY`. The `@Pattern(regexp = "ALLOW|DENY")` constraint on the entity would cause a validation error — or worse, silent storage of an invalid value.
**Why it happens:** `SecPermission` uses `GRANT`/`DENY` while `SecMenuPermission` uses `ALLOW`/`DENY`. The `CurrentUserMenuPermissionService` already filters on `"ALLOW"`. There is no `SecPermissionUiContractService` normalization for menu permissions.
**How to avoid:** The admin frontend for menu permissions must send `effect: 'ALLOW'` directly. The `AdminMenuPermissionResource` stores and returns `ALLOW`/`DENY` without transformation. Document this in the DTO.
**Warning signs:** If unit tests or integration tests see 400 responses from the `@Valid` constraint on the `effect` field.

### Pitfall 2: Missing delete cascade for SecMenuPermission on definition delete
**What goes wrong:** Deleting a `SecMenuDefinition` row leaves orphaned `SecMenuPermission` rows pointing to the deleted `menuId`. These orphans are invisible to admins and can interfere with future role permission auditing.
**Why it happens:** There is no DB-level foreign key from `sec_menu_permission.menu_id` to `sec_menu_definition.menu_id` (the FK only exists from `sec_menu_permission.role` to `jhi_authority.name`).
**How to avoid:** Add `deleteByAppNameAndMenuId(String appName, String menuId)` to `SecMenuPermissionRepository` and call it in a `@Transactional` delete handler in `SecMenuDefinitionAdminResource`.
**Warning signs:** Growing `sec_menu_permission` row count after repeated create/delete cycles in the admin UI.

### Pitfall 3: PrimeNG TabsModule vs TabViewModule naming
**What goes wrong:** PrimeNG v19 uses `TabsModule` (with `p-tabs`, `p-tab-list`, `p-tab`, `p-tab-panels`, `p-tab-panel` components) not the older `TabViewModule` (with `p-tabView` and `p-tabPanel`). Importing the wrong module causes template parse errors.
**Why it happens:** The UI-SPEC says `TabsModule` but developers familiar with PrimeNG v17 may reach for `TabViewModule`.
**How to avoid:** Import `TabsModule` from `primeng/tabs`. Use `<p-tabs>` + `<p-tab-list>` + `<p-tab>` + `<p-tab-panels>` + `<p-tab-panel>` syntax.
**Warning signs:** Template compilation error `'p-tabView' is not a known element`.

### Pitfall 4: Tree selection propagation overrides deny-default intent
**What goes wrong:** PrimeNG `p-tree` with `selectionMode="checkbox"` defaults to propagating selection up and down the hierarchy. Checking a parent node automatically checks all children. This conflicts with D-07 (each node independently granted — no implicit propagation).
**Why it happens:** PrimeNG default behavior for tree checkbox selection is parent→child propagation.
**How to avoid:** Set `propagateSelectionUp="false"` and `propagateSelectionDown="false"` on `p-tree`. Map checked state independently for each leaf node. The UI-SPEC already prescribes this.
**Warning signs:** Checking "Security" section node grants all security children silently.

### Pitfall 5: Navigation registry node for menu-definitions missing from security guard
**What goes wrong:** Adding the `/admin/security/menu-definitions` route without adding `security.menu-definitions` to `APP_NAVIGATION_TREE` means `UserRouteAccessService` cannot find the node and the route access guard may behave unexpectedly.
**Why it happens:** The route data includes `navigationNodeId` which the guard looks up in the navigation registry. Missing entries fail silently or deny access.
**How to avoid:** Add a `security.menu-definitions` leaf node to the `security` section in `navigation-registry.ts` alongside `security.row-policies`.
**Warning signs:** Navigating to `/admin/security/menu-definitions` redirects to access denied even as an admin.

### Pitfall 6: `i18n` keys missing for new UI strings
**What goes wrong:** New components reference translation keys (`pageTitle.menuDefinitions`, `layout.menu.security.menuDefinitions`, `feedback.security.menuDefinitions.*`) that do not exist in `en.json` and `vi.json`. The translate pipe renders the raw key string.
**Why it happens:** Translation keys must be manually added; there is no auto-discovery.
**How to avoid:** Add all keys to both `frontend/public/i18n/en.json` and `frontend/public/i18n/vi.json` as part of the implementation wave that introduces the new components.
**Warning signs:** Admin UI shows raw key strings like `pageTitle.menuDefinitions` instead of "Menu Definitions".

### Backend: Sync endpoint design

```java
// POST /api/admin/sec/menu-definitions/sync
@PostMapping("/sync")
@Transactional
public ResponseEntity<SyncResultDTO> syncFromRegistry(@Valid @RequestBody SyncRequestDTO request) {
    // request.nodes: List of { menuId, appName, menuName, label, parentMenuId, route, icon, ordering }
    int seeded = 0;
    for (SyncNodeDTO node : request.getNodes()) {
        if (menuDefinitionRepository.findByAppNameAndMenuId(node.getAppName(), node.getMenuId()).isEmpty()) {
            // insert new definition
            seeded++;
        }
        // skip existing — idempotent
    }
    return ResponseEntity.ok(new SyncResultDTO(seeded, request.getNodes().size() - seeded));
}
```

## Open Questions

1. **Sync endpoint: who sends the registry data?**
   - What we know: the backend has no native access to the Angular `APP_NAVIGATION_TREE` at runtime.
   - What's unclear: should the frontend send the full registry snapshot in the POST body, or should there be a static YAML/properties file listing node IDs?
   - Recommendation: frontend sends the registry snapshot in the POST body (simplest; no backend YAML coupling; already done at the call site). The planner should model this as `POST /sync` with a `nodes[]` array body.

2. **`SecMenuDefinitionAdminResource` — should it filter by appName or return all?**
   - What we know: `SecMenuDefinitionRepository.findAllByAppName(appName)` exists; `findAll()` also exists.
   - What's unclear: does the admin list page need to show definitions from multiple apps?
   - Recommendation: the list endpoint should accept an optional `?appName=` query parameter; default to `jhipster-security-platform` when absent. This matches D-09 scope (current phase targets one app only) while leaving multi-app flexibility.

3. **`AdminMenuPermissionResource` — bulk assign vs single record?**
   - What we know: the existing `SecPermissionAdminResource` is single-record. The Permission Matrix flushes individual create/delete operations via `forkJoin`.
   - What's unclear: the tree checkbox UI may produce many simultaneous grant changes.
   - Recommendation: single-record pattern (POST one permission, DELETE one permission by ID) is consistent with the existing matrix pattern and requires no new bulk endpoint.

## Project Constraints (from CLAUDE.md)

These CLAUDE.md directives are binding on all implementation in this phase:

| Directive | Impact on This Phase |
|-----------|---------------------|
| `standalone: true` must NOT be set in Angular decorators (default in v20+) | Do not add `standalone: true` to any new component decorator |
| Use `input()` / `output()` functions, not `@Input()` / `@Output()` decorators | New signal-based components use `input()`/`output()`. Dialog components that follow the role-dialog pattern (which uses `@Input`) should match that pattern for consistency — note the tension here; prefer `input()`/`output()` for any net-new component that does not need to exactly clone the role-dialog API |
| `ChangeDetectionStrategy.OnPush` required | Set on all new components |
| No `ngClass` — use `class` bindings | Conditional CSS must use `[class.xxx]="condition"` |
| No `ngStyle` — use `style` bindings | Inline styles use `[style.xxx]="value"` |
| Native control flow (`@if`, `@for`, `@switch`) only | No `*ngIf` / `*ngFor` in new templates |
| `inject()` function for DI in Angular components and services | No constructor injection in Angular code |
| `ApplicationConfigService.getEndpointFor(...)` for all API URL resolution | Do not hardcode `/api/...` prefixes in services |
| WCAG AA + AXE pass required | Tree checkboxes need visible labels; all form inputs need explicit `<label for>` binding |
| `aria-live="polite"` for unsaved changes count | Wrap pending change count in `aria-live="polite"` span |
| ArchUnit layer rules: `web` → `service` → `repository` | New backend resource may not reach directly into repository without a service layer if business logic is involved; for simple CRUD the existing pattern (direct repo access in admin resource) is acceptable per SecRoleAdminResource precedent |
| `LOG` for SLF4J logger name | Backend resources declare `private static final Logger LOG = LoggerFactory.getLogger(...)` |
| Throw `BadRequestAlertException` for domain violations | Duplicate menuId per appName, role not found, ID mismatch |
| `@PreAuthorize("hasAuthority('ROLE_ADMIN')")` on new admin resources | Both `SecMenuDefinitionAdminResource` and `AdminMenuPermissionResource` |
