## Spacing Scale

Declared values (multiples of 4, aligned with Tailwind utility classes in use across existing admin components):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap (`gap-1`), inline badge padding |
| sm | 8px | Button group gap (`gap-2`), form field help-text gap |
| md | 16px | Form field vertical gap (`gap-4`), card interior padding |
| lg | 24px | Card header padding (`px-4 pt-4` = 16px, section margin `mb-6`) |
| xl | 32px | Splitter panel padding, major section vertical rhythm |
| 2xl | 48px | Empty-state vertical padding (`py-8 = 32px` used in existing; 48px for full-height empty states) |
| 3xl | 64px | Page-level top/bottom padding (not used in admin shell; reserved) |

Exceptions:
- Touch targets for action buttons: minimum 36px height via PrimeNG `size="small"` button (inherits from PrimeNG design token `--p-button-sm-*`).
- Dialog width: 480px (matching role-dialog pattern, source: `role-dialog.component.html`).
- Tree node rows: 40px minimum height for click target accessibility.

## Screen Layouts

### Layout 1: Menu Definitions List

```
┌─────────────────────────────────────────────────────┐
│ [p-card]                                            │
│  header: "Menu Definitions"    [Sync from Registry] │
│                                  [New Menu Definition]│
│ ┌─────────────────────────────────────────────────┐ │
│ │ p-table                                         │ │
│ │ Menu ID | Label | Parent | Route | Icon | Order │ │
│ │ ─────── | ───── | ────── | ───── | ──── | ───── │ │
│ │ row...  |       |        |       |      | [Edit][Delete] │
│ └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

Table columns (in order): Menu ID, Label, Parent Menu ID (shows "—" if null), Route (truncated at 40 chars), Icon, Ordering, Actions.
Actions column: "Edit" (`secondary small`) + "Delete" (`danger outlined small`).

### Layout 2: Menu Access Tab (on existing Permission Matrix page)

The existing `p-splitter` layout is replaced by `p-tabs` when this tab is introduced. The Entity Permissions panel becomes Tab 1, the Menu Access panel becomes Tab 2.

```
┌──────────────────────────────────────────────────────────┐
│ [← Back to Roles]   Permissions for {roleName}          │
│ [Save Changes] [Discard]  {N unsaved changes}           │
│                                                          │
│ [Entity Permissions] [Menu Access]  ← p-tabs            │
│                                                          │
│ Tab 2 active:                                            │
│ ┌─────────────────────────────────────────────────────┐  │
│ │ p-tree (selection mode: checkbox)                   │  │
│ │ ▼ Home                                              │  │
│ │   □ Dashboard                                       │  │
│ │ ▼ Entities                                          │  │
│ │   □ Organization                                    │  │
│ │   □ Department                                      │  │
│ │   □ Employee                                        │  │
│ │ ▼ Security                                          │  │
│ │   □ Users                                           │  │
│ │   □ Roles                                           │  │
│ │   □ Row Policies                                    │  │
│ │   □ Menu Definitions                                │  │
│ └─────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

Tree data source: `APP_NAVIGATION_TREE` from `navigation-registry.ts` mapped into PrimeNG `TreeNode[]` objects. Checked state represents whether a `SecMenuPermission` row exists for `role + menuId`.

Tree selection mode: `checkbox` (PrimeNG p-tree `selectionMode="checkbox"` with `propagateSelectionUp` and `propagateSelectionDown` both false — each node is independently granted, matching the deny-default D-07 decision).

### Dialog Layout: Create/Edit Menu Definition

```
┌─────────────────────────────────────────────────────┐
│ New Menu Definition / Edit Menu Definition          │
├─────────────────────────────────────────────────────┤
│ Menu ID *         [_______________________________] │
│ Menu Name *       [_______________________________] │
│ Label *           [_______________________________] │
│ Parent Menu ID    [_______________________________] │
│ Route             [_______________________________] │
│ Icon              [_______________________________] │
│ Ordering *        [___]                             │
│ Description       [                               ] │
```

## Interaction States

### Menu Definitions List

| State | Behavior |
|-------|---------|
| Loading | `[loading]="isLoading"` on p-table shows PrimeNG built-in skeleton rows |
| Empty | `#emptymessage` template: icon `pi pi-list text-4xl`, heading "No menu definitions", body "Create the first menu definition or sync from the registry." |
| Row selected for delete | `confirmDelete()` opens p-confirmDialog before DELETE fires |
| Sync in progress | "Sync from Registry" button shows `[loading]="isSyncing"` spinner; disables button |

### Menu Access Tab

| State | Behavior |
|-------|---------|
| Loading | Full-tab `p-progressSpinner` centered in tab panel body |
| No pending changes | "Save Changes" and "Discard" buttons are `[disabled]="true"` |
| Pending changes | Orange indicator count "N unsaved change(s)" appears; buttons enabled |
| Save in progress | `[loading]="saving"` on Save button; checkboxes `[disabled]="saving"` |
| Empty permissions (new role) | Tree renders with all nodes unchecked; no special empty state needed |

### Dialog

| State | Behavior |
|-------|---------|
| Invalid form on submit | `save()` returns early; required field labels turn red (PrimeNG invalid class); error hint appears below field |
| Saving | Save button `[loading]="isSaving"` spinner; form disabled |
| Save success | Dialog closes; list reloads via `onSaved()` event; success toast appears |
| Save error | Dialog stays open; error toast appears via `handleHttpError` |

## Accessibility Requirements

These requirements apply in addition to the WCAG AA minimums declared in CLAUDE.md.

- All form inputs must have explicit `<label for="...">` bindings (matching `role-dialog.component.html` pattern).
- Tree checkboxes must have visible labels (node label text) so screen readers identify the grant target.
- Action buttons must have `aria-label` attributes when the label is icon-only. All buttons in this phase have visible text labels — no icon-only buttons are used.
- Focus must return to the triggering button when a dialog closes (PrimeNG Dialog handles this automatically with `focusOnShow`).
- "N unsaved changes" live region: wrap in `aria-live="polite"` `<span>` so change count is announced without interrupting the user.
- Destructive confirm dialog must place focus on the "Keep Definition" reject button by default (safer default, prevents accidental delete via Enter key). Implement via PrimeNG `ConfirmationService` `defaultFocus: 'reject'`.
