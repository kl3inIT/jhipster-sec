Add `SecPermissionUiContractService.java` that builds generic normalization maps from `SecuredEntityCatalog.entries()` plus JPA metamodel data instead of hard-coded entity names.

Use this exact algorithm:
- For each catalog entry, derive entity UI target from `entry.code()` and stored entity target from `entry.entityClass().getSimpleName().toUpperCase(Locale.ROOT)`.
- Enumerate metamodel attributes for each `entry.entityClass()` and derive attribute mappings for every `entry.code() + "." + attributeName`.
- Add wildcard mappings for every secured entity: `entry.code() + ".*"` -> `SIMPLECLASSNAME.*`.
- Build both directions: UI -> stored and stored -> UI.
- Apply normalization only for `targetType` values `ENTITY` and `ATTRIBUTE`; leave other target types unchanged.

Normalize effects generically:
- Incoming UI `GRANT` must be stored as `ALLOW`.
- Incoming `ALLOW` and `DENY` remain valid for backward-compatible admin callers.
- Outgoing stored `ALLOW` must be translated back to UI `GRANT` on `GET /api/admin/sec/permissions` and `GET /api/admin/sec/permissions/{id}` so D-23/D-24 stay intact.

Modify `SecPermissionAdminResource.java` so create/update paths translate incoming DTOs before save, and get/getAll paths translate stored DTOs back to the UI contract.

Extend `SecPermissionAdminResourceIT.java` with the three matrix-style round-trip cases above, including the second-entity/second-attribute case `employee.salary`.
