# Phase 8.1: Jmix-Style DataManager Core Alignment - Research

## User Constraints (from CONTEXT.md)

### Locked Decisions
- `SecureDataManager` stays the application-facing facade for protected services and frontend/API-facing CRUD flows.
- A new internal `DataManager` must extend `UnconstrainedDataManager`, with secure-by-default behavior and `unconstrained()` as the explicit bypass.
- Raw repository mechanics move into `UnconstrainedDataManagerImpl`; authorization and registered access constraints move into `DataManagerImpl`.
- Resource permissions move to default-deny plus union-of-`ALLOW` semantics for entity, attribute, menu, and capability-style resource evaluation.
- Brownfield auth/account/admin/mail flows and the existing map/fetch-plan/entity-code contract are protected behavior for this phase.

### User Override on 2026-03-26
- Row-policy refactoring is temporarily out of scope for Phase 08.1.
- The plan may keep existing row-policy wiring compiling, but it must not make row-policy redesign or dedicated row-policy verification a delivery objective in this run.

### Agent's Discretion
- The exact method set on `DataManager`, as long as it clearly separates secure operations from the explicit bypass.
- Whether wildcard attribute targets are handled by one repository query or by combining exact and wildcard lookups in the evaluator.
- Whether brownfield verification extends existing integration tests or adds one focused new regression suite.

### Deferred Ideas (OUT OF SCOPE)
- Replacing the current `SecureDataManager` map payload boundary with typed DTO/entity returns.
- Database-managed fetch plans or runtime fetch-plan editing.
- Row-policy redesign, row-policy cleanup, or row-policy-specific verification expansion.

## Phase Requirements

| ID | Requirement |
|----|-------------|
| SC-01 | Add an internal `DataManager` abstraction that extends `UnconstrainedDataManager` and exposes `unconstrained()` |
| SC-02 | Reuse `UnconstrainedDataManagerImpl` as the mechanical core while `DataManagerImpl` becomes the secure default path |
| SC-03 | Keep `SecureDataManager` as the stable application-facing facade and do not break protected service or frontend/API contracts |
| SC-04 | Change entity, attribute, menu, and capability layers to default-deny plus union-of-`ALLOW` semantics |
| SC-05 | Keep the explicit unconstrained path narrow and intentional |
| SC-06 | Prove the new behavior with focused unit/integration coverage and rerun brownfield auth/account/admin safety checks |

## Summary

The current root implementation already has both ends of the desired split, but they are arranged incorrectly. `SecureDataManagerImpl` currently owns CRUD authorization, repository querying, merge orchestration, fetch-plan resolution, and secure serialization in one class, while `UnconstrainedDataManagerImpl` is only a thin bypass and is not the shared mechanical core beneath the secure path.

`angapp` is useful as a donor for boundary shape, not as a literal implementation to copy. Its application services still depend on a single secured facade, which supports keeping `SecureDataManager` stable, but it does not provide the internal `DataManager` / `UnconstrainedDataManager` split this phase wants. Its permission service also still uses deny-wins semantics, so Phase 08.1 is an intentional alignment step rather than a straight port.

The safest delivery sequence is:
1. Expand the unconstrained mechanical API and introduce `DataManagerImpl` as the internal secure seam.
2. Migrate every resource-permission hot spot together so semantics do not diverge by subsystem.
3. Prove the new seam does not break protected entity CRUD or legacy JHipster auth/admin endpoints.

## Standard Stack

### Core
- Keep Java 21, Spring Boot 4, Spring Data JPA, and the existing repository-registry / access-manager / fetch-plan stack.
- Keep `SecureDataManager`, `SecureMergeService`, `SecureEntitySerializer`, and `FetchPlanResolver` as the protected-boundary orchestration layer.
- Keep the current `MergedSecurityService` and database-backed `SecPermission` / `SecMenuPermission` storage model.

### Supporting
- No new runtime dependency is required for this phase.
- Unit coverage stays on JUnit 5 + Mockito.
- Integration coverage stays on the existing `@IntegrationTest` + `MockMvc` setup.

### Alternatives Considered
- Replacing `SecureDataManager` with `DataManager` across application services now.
  Rejected: it risks frontend/API contract drift and violates the locked boundary-preservation decision.
- Copying `angapp`'s `SecureDataManagerImpl` directly.
  Rejected: it still keeps secure orchestration and raw persistence coupled and does not deliver the requested internal split.
- Changing only `RolePermissionServiceDbImpl` and leaving menu/capability/attribute handling for later.
  Rejected: it would leave mixed permission semantics across the same user session.

## Architecture Patterns

### Pattern 1: Make `UnconstrainedDataManagerImpl` the actual mechanical core
`UnconstrainedDataManagerImpl` should own raw repository-backed load/save/delete/query mechanics, not just the current `load(id)` / `loadAll()` / `save(entity)` helpers.

Concrete planning target:
- keep existing methods
- add spec-aware mechanics the secure layer can reuse
- keep these methods free of CRUD checks, fetch-plan work, secure serialization, and merge logic

### Pattern 2: Add `DataManagerImpl` as the secure-default internal seam
`DataManagerImpl` should centralize:
- CRUD permission checks through `AccessManager` and `CrudEntityContext`
- secure lookup helpers for READ / UPDATE / DELETE paths
- composition with the shared raw mechanics from `UnconstrainedDataManagerImpl`
- the explicit bypass handle returned by `unconstrained()`

### Pattern 3: Keep `SecureDataManagerImpl` as boundary orchestration, not raw persistence
After the split, `SecureDataManagerImpl` should still own:
- secured entity-code resolution from `SecuredEntityCatalog`
- fetch-plan resolution and serialization to `Map<String, Object>`
- secure merge of map payloads onto entities
- protected-boundary entrypoints used by `OrganizationService`, `DepartmentService`, and `EmployeeService`

It should stop reaching directly into `RepositoryRegistry` for its secure path mechanics.

### Pattern 4: Migrate resource permission semantics as one sweep
The current deny-wins behavior is spread across:
- `RolePermissionServiceDbImpl`
- `AttributePermissionEvaluatorImpl`
- `CurrentUserMenuPermissionService`
- `SecuredEntityCapabilityService.PermissionMatrix`

These must move together to the same rule:
- empty match set -> denied
- one or more `ALLOW` records -> allowed
- a `DENY` record does not override another role's `ALLOW` in this phase

This phase should also align wildcard attribute handling between live evaluation and capability payload generation so the UI and enforcement pipeline compute the same result.

### Pattern 5: Keep the bypass explicit and narrow
The new `unconstrained()` path is necessary, but it should stay explicit in call sites. New unconstrained usage outside the security data layer or focused tests should be treated as suspicious by default.

### Anti-Patterns to Avoid
- Leaving `SecureDataManagerImpl` with both `RepositoryRegistry` orchestration and the new `DataManager` injection.
- Updating menu or capability semantics without updating entity/attribute evaluation in the same phase.
- Moving fetch-plan serialization into `DataManagerImpl`; that would collapse the protected boundary into the low-level secure core.
- Hiding the bypass path behind implicit fallback logic instead of the explicit `unconstrained()` call.
- Replacing deny-wins tests without adding positive multi-role union coverage.

## Don't Hand-Roll
- Do not add a new persistence abstraction below `RepositoryRegistry`; the existing registry is already the runtime repository switchboard.
- Do not create new database metadata for fetch plans or permission merge modes.
- Do not invent a second secure facade for protected application services; keep `SecureDataManager`.

## Common Pitfalls

### Pitfall 1: `DataManagerImpl` duplicates repository logic instead of delegating to `UnconstrainedDataManagerImpl`
That would preserve the current coupling under a new class name and fail SC-02.

### Pitfall 2: CREATE checks regress because the new secure helpers only model lookup paths
The plan must keep an explicit CREATE permission check path before entity instantiation.

### Pitfall 3: Attribute enforcement and capability payloads drift
`AttributePermissionEvaluatorImpl` and `SecuredEntityCapabilityService.PermissionMatrix` must agree on both wildcard handling and union-of-`ALLOW` semantics.

### Pitfall 4: Protected services accidentally start depending on `DataManager`
That would widen the internal refactor into an application contract change and increase brownfield risk.
