## Tests

### 1. Menu Visibility Uses Union Of Allow
expected: For a user who has multiple roles, any menu with at least one matching ALLOW should still appear in the navigation or in the `/api/security/menu-permissions?appName=jhipster-security-platform` response, even if another role contributes a DENY for that same menu. Menus with only DENY and no ALLOW should stay absent.
result: pass

### 2. Capability Payload Matches Allowed And Denied Attributes
expected: `/api/security/entity-capabilities` and any frontend screens driven by it should agree on CRUD and field access. Reader-style access should expose read-only organization capability, editor-style access should expose create/update/delete where granted, and denied attributes such as `budget` should remain non-viewable or non-editable.
result: pass

### 3. Organization List Respects Row And Attribute Rules
expected: Opening the protected organization list or calling `/api/organizations?sort=id,asc` should return only row-allowed records. Allowed fields such as `code` and `name` should be present, while denied fields such as `budget` should be omitted from the payload or UI.
result: pass

### 4. Organization Detail Preserves Nested Secure Reads
expected: Opening an allowed organization detail should still include nested departments and employees from the secure fetch-plan path. Visible nested fields such as department names and employee names should be present, while denied fields such as organization `budget` and employee `salary` should remain hidden.
result: pass

### 5. No Allow Still Denies But Any Allow Unlocks Access
expected: A user with no organization READ allow should still be blocked from `/api/organizations`. After granting an ALLOW on one of the user's authorities, the same user should be able to load the allowed owned data, even if another authority on that user still carries a DENY.
result: pass

### 6. Editor Can Create And Update Allowed Fields
expected: With editor-level organization permissions, creating a new organization should succeed and the new record should appear in the organization list. Updating an allowed field such as `name` should persist and be visible on the follow-up read.
result: pass

### 7. Denied Fields And Out-Of-Scope Rows Stay Blocked
expected: Attempting to edit a denied field such as `budget` should fail or the field should not be offered for editing. Attempting to read or delete an organization outside the caller's row-policy scope should still fail rather than leaking access after the DataManager split.
result: pass

### 8. Brownfield Account And Admin Flows Still Hold
expected: Existing auth/account and admin-user flows should continue to behave as before. `/api/account` should still return the current user, account update and password-change flows should still work, and admin user list/detail/create/update/delete behavior should remain stable for admin users.
result: pass
