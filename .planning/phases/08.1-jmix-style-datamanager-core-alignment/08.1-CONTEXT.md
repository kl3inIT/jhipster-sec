# Phase 8.1: Jmix-Style DataManager Core Alignment - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Align the internal data-access architecture with a Jmix-style secure-default path without changing the current application-facing `SecureDataManager` contract used by services and the frontend/API boundary. This phase introduces a new internal `DataManager` abstraction that extends `UnconstrainedDataManager`, moves raw data mechanics into `UnconstrainedDataManagerImpl`, and centralizes authorization plus registered access constraints in `DataManagerImpl`. It does not change auth/account/admin/mail flows, it does not introduce database-backed fetch-plan definitions, and it does not remove the brownfield map/fetch-plan/entity-code contract that current application services already rely on.

**Outcome**
- Protected application services keep using `SecureDataManager`.
- Internal secured reads and writes flow through `DataManager` by default.
- `unconstrained()` becomes the explicit bypass path.
- Resource permission semantics align to default deny plus union-of-`ALLOW`, while row-level policies stay a separate enforcement layer.

</domain>

<decisions>
## Implementation Decisions

### Boundary Preservation
- **D-01:** `com.vn.core.security.data.SecureDataManager` remains the stable application-facing facade for `OrganizationService`, `DepartmentService`, `EmployeeService`, and any frontend/API-driven workflows.
- **D-02:** Introduce an internal `DataManager` abstraction that extends `UnconstrainedDataManager`; the secure path is the default and `unconstrained()` is the explicit escape hatch.
- **D-03:** `SecureDataManagerImpl` keeps app-specific responsibilities only: entity-code resolution, fetch-plan-driven serialization, map payload handling, secure merge orchestration, and protected-boundary row-policy orchestration.

### Internal Layering
- **D-04:** Raw CRUD/query/load/save/delete mechanics move into `UnconstrainedDataManagerImpl`; `DataManagerImpl` composes those mechanics instead of reimplementing repository interaction itself.
- **D-05:** Authorization and registered access-constraint application move into `DataManagerImpl` so secure enforcement is centralized instead of being duplicated across callers.
- **D-06:** Row-level policies remain a separate enforcement layer in the secure path and are not merged into resource-permission resolution.

### Permission Semantics
- **D-07:** Resource-role permissions use default deny plus union-of-`ALLOW` semantics for entity, attribute, menu, and similar resource permissions; no global deny-wins merge remains unless a specific brownfield deviation is explicitly documented.
- **D-08:** Any behavior that must remain different from Jmix for brownfield safety must be called out in the implementation docs and verification output as an intentional deviation.
- **D-09:** Existing auth/account/admin/mail and frontend contract behavior are protection targets for this phase, not refactor collateral.

### Suggested Delivery Split
- **D-10:** Plan this phase as at least three substeps: architecture seam introduction, permission-semantic refactor, and regression/verification.

### the agent's Discretion
- Exact interface method set on the new internal `DataManager`, as long as it clearly extends `UnconstrainedDataManager` and exposes `unconstrained()`
- Exact naming of helper classes that consolidate resource permission merging, as long as semantics and layering are explicit
- Exact test split between unit and integration coverage, as long as the acceptance cases are proven end-to-end

</decisions>

<code_context>
## Existing Code Insights

### Current Layering Reality
- `SecureDataManagerImpl` currently resolves catalog entry, performs CRUD authorization, builds row specifications, executes repository queries, resolves fetch plans, serializes secure payloads, performs merge, and persists entities directly. The secure orchestration and the raw mechanics are tightly coupled in one class.
- `UnconstrainedDataManagerImpl` is already the trusted bypass path, but it only exposes simple load/loadAll/save/delete mechanics and is not the reused core beneath the secured path.
- There is no internal `DataManager` abstraction today. Application services call `SecureDataManager` directly, and `SecureDataManagerImpl` reaches the repositories without an intermediate secured internal layer.

### Current Permission-Semantic Reality
- `RolePermissionServiceDbImpl` currently returns `false` if any matching `DENY` record exists, even if another role `ALLOW`s the same entity/action.
- `AttributePermissionEvaluatorImpl` currently treats empty attribute permissions as denied and also uses deny-wins when rules exist.
- `CurrentUserMenuPermissionService` currently builds a denied-menu set first and strips those menu ids out of the final allow list, so one deny from any role blocks a menu even when another role allows it.
- `SecuredEntityCapabilityService.PermissionMatrix` currently hard-codes deny-wins rules for entity and attribute capability calculation, which means frontend gating payloads will also need coordinated refactor coverage.

### Brownfield Stability Constraints
- `OrganizationService`, `DepartmentService`, and `EmployeeService` already depend only on `SecureDataManager`, so the safest refactor keeps that facade stable.
- Row policies already have a separate provider and specification builder; the refactor should preserve that separation instead of folding row logic into role/resource permission merging.
- Fetch-plan resolution, map payload shaping, and secure merge are application-facing concerns already handled inside `SecureDataManagerImpl`; moving them wholesale into a low-level core would blur the boundary the project currently relies on.

### Test Impact
- Existing unit tests explicitly describe current deny-wins semantics for role and menu permissions.
- New tests will need to prove multi-role union-of-`ALLOW` behavior while retaining row-level fail-closed behavior and the explicit unconstrained bypass path.

</code_context>

<risks>
## Risks And Guardrails

- **Semantic drift risk:** Changing only one evaluator or service will leave mixed permission semantics between backend authorization, menu visibility, and capability payloads.
- **Boundary regression risk:** If `SecureDataManager` loses its map/fetch-plan/entity-code responsibilities, protected entity services and frontend contracts can break even if lower-level data access becomes cleaner.
- **Bypass misuse risk:** Adding `unconstrained()` without keeping it explicit and limited could make security bypasses easy to call accidentally.
- **Row-policy regression risk:** Centralizing authorization must not accidentally remove row-policy filtering from secured reads, writes, or deletes.
- **Test gap risk:** Existing deny-wins tests will need intentional replacement, not simple deletion, so the new semantics are proven rather than assumed.

</risks>

<acceptance>
## Acceptance Criteria

- There is a new internal `DataManager` abstraction that extends `UnconstrainedDataManager`.
- `DataManagerImpl` reuses `UnconstrainedDataManagerImpl` mechanics in a Jmix-like secure-default design.
- The unconstrained path is explicit via `unconstrained()`.
- `SecureDataManager` remains the stable application-facing facade.
- Resource permission merging changes to default deny plus union-of-`ALLOW` for entity, attribute, menu, and similar resource permissions unless an intentional deviation is documented.
- Row-level restrictions still apply correctly in the protected data-access path.
- Tests prove multiple roles accumulate `ALLOW` permissions, menu/resource permissions are no longer blocked by a global deny-wins strategy, row-level restrictions still apply, and unconstrained access bypasses authorization by design.
- The phase output documents which pieces are Jmix-aligned and which remain intentionally project-specific for brownfield safety.

</acceptance>

<deferred>
## Deferred Ideas

- Public API contract simplification beyond the existing `SecureDataManager` map/fetch-plan/entity-code boundary
- Database-backed fetch-plan authoring or runtime fetch-plan administration
- Broader security-model redesign outside resource-permission semantics and secure data-access layering

</deferred>
