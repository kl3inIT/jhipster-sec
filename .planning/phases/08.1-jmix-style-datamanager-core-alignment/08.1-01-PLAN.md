  <action>
    Create `DataManager.java` in `com.vn.core.security.data` with this exact contract:
    - `public interface DataManager extends UnconstrainedDataManager`
    - `UnconstrainedDataManager unconstrained();`
    - `<T> Optional<T> loadOne(Class<T> entityClass, Specification<T> spec, EntityOp op);`
    - `<T> List<T> loadList(Class<T> entityClass, Specification<T> spec, EntityOp op);`
    - `<T> Page<T> loadPage(Class<T> entityClass, Specification<T> spec, Pageable pageable, EntityOp op);`
    - `void checkCrud(Class<?> entityClass, EntityOp op);`

    Create `DataManagerImpl.java` as a `@Service` that injects `AccessManager` and `UnconstrainedDataManager`.
    - `checkCrud(...)` must build `CrudEntityContext`, call `accessManager.applyRegisteredConstraints(...)`, and throw `AccessDeniedException` when `ctx.isPermitted()` is false
    - `loadOne(..., spec, op)`, `loadList(..., spec, op)`, and `loadPage(..., spec, pageable, op)` must call `checkCrud(...)` first, then delegate to `unconstrainedDataManager`
    - `unconstrained()` must return the injected `UnconstrainedDataManager`
    - do not move `RowLevelSpecificationBuilder` into `DataManagerImpl` in this phase

    Update `SecureDataManagerImpl.java` so:
    - it injects `DataManager` instead of `AccessManager` and `RepositoryRegistry`
    - it keeps the existing `RowLevelSpecificationBuilder` injection and current row-spec composition untouched
    - `loadList` builds the current row spec locally, then calls `dataManager.loadPage(entityClass, rowSpec, pageable, EntityOp.READ)`
    - `loadByQuery` builds the current row spec locally and calls `dataManager.loadPage(entityClass, finalSpec, query.pageable(), EntityOp.READ)` after the existing JPQL guard
    - `loadOne` builds the current id+row spec locally and calls `dataManager.loadOne(entityClass, idSpec.and(rowSpec), EntityOp.READ)`
    - `save` uses `dataManager.checkCrud(entityClass, EntityOp.CREATE)` and `dataManager.unconstrained().create(entityClass)` for creates
    - `save` uses `dataManager.loadOne(entityClass, idSpec.and(rowSpec), EntityOp.UPDATE)` for update target lookup, then persists through `dataManager.unconstrained().save(entity)`
    - `delete` uses `dataManager.loadOne(entityClass, idSpec.and(rowSpec), EntityOp.DELETE)` then `dataManager.unconstrained().delete(entity)`
    - `SecureDataManagerImpl` no longer imports or references `RepositoryRegistry`

    Add `DataManagerImplTest.java` to prove:
    - CRUD denial throws before unconstrained access
    - READ delegates to `unconstrainedDataManager.loadPage`
    - `unconstrained()` returns the injected bypass object

    Update `SecureDataManagerImplTest.java` so the ordering assertions verify delegation to `DataManager` / `UnconstrainedDataManager` instead of direct repository interaction.
