## Jmix-aligned

- Internal secured access now flows through a `DataManager` / `UnconstrainedDataManager` split instead of mixing secure orchestration and raw repository mechanics in one class.
- The secure path is the default path and the bypass path is explicit through `DataManager.unconstrained()`.
- Resource permissions now follow default-deny plus union-of-`ALLOW` semantics across entity, attribute, menu, and capability answers.
- Menu visibility no longer uses a global deny-wins merge; any matching `ALLOW` can surface the menu.

## Intentionally project-specific

- `SecureDataManager` remains the stable application-facing facade for protected services and frontend/API workflows.
- Fetch-plan-driven secure reads still serialize to `Map<String, Object>` payloads instead of exposing a Jmix-style application API directly.
- Fetch plans remain YAML/code-defined only; database-backed fetch-plan metadata is still intentionally out of scope.
- Row policies remain a separate enforcement layer composed into the protected boundary rather than merged into resource-permission evaluation.
