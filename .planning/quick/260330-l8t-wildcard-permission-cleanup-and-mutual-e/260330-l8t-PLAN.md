**Component TS changes (`permission-matrix.component.ts`):**

1. Replace the simple `togglePermission` method with smart mutual-exclusion logic. Rename the existing method to `_applySimpleToggle` (private) and create a new `togglePermission` that delegates:

   a) **Entity permission toggle** (`targetType === 'ENTITY'`):

      - **Toggling wildcard `*` ON**: Set pending `*:action = checked:true`. For every catalog entry code, if that specific entity has a pending-add, delete it (it becomes redundant). No need to mark specifics as pending-delete here since the backend cleanup handles it on save. But we DO need to mark any currently-granted specific entities as pending-delete so the flush will delete them. Iterate `this.catalogEntries`, for each entry code, if `this.granted.has(permissionKey(code, action))` and no pending already, set `pendingChanges` for that code+action to `{checked: false, targetType: 'ENTITY', target: code, action}`. If a pending-add exists for a specific, just delete it.

      - **Toggling wildcard `*` OFF**: Set pending `*:action = checked:false`. Remove all pending-delete entries for specific entities that were auto-generated by the wildcard-on logic (they were only pending because wildcard was being added).

      - **Toggling specific entity ON while `*` is effectively granted**: This is the "untick wildcard by adding all others" flow, but since user is ADDING a specific while wildcard is on, this is a no-op visually (already checked). Actually per the task description, checkboxes should NOT be disabled, so user can UNTICK a specific while wildcard is on. Handle the untick case next.

      - **Toggling specific entity OFF while `*` is effectively granted** (the key interaction):
        (i) Mark `*:action` as pending unchecked (remove wildcard).
        (ii) For every OTHER catalog entry code (not the one being unticked), add a pending-add: `{checked: true, targetType: 'ENTITY', target: otherCode, action}` (unless it is already granted without pending).
        (iii) The unticked entity itself: if it was granted in DB, mark pending-delete; if it was only effectively granted via wildcard, no pending needed (it just won't be added).
        (iv) Clean up: if a specific was already granted in DB AND wildcard was also granted in DB, the specific is redundant and should not get a pending-add (it already exists). So only add pending-add for entries that are NOT already in `this.granted`.

      - **All specifics now checked (auto-promote to wildcard)**: After any specific entity toggle-ON, check if ALL catalog entry codes are now effectively granted (via granted + pending). If so, auto-promote: set `*:action` as pending-add, and for each specific that is granted in DB, set pending-delete; for each specific that is pending-add, remove the pending (wildcard covers it).

   b) **Attribute permission toggle** (`targetType === 'ATTRIBUTE'`): Same mutual-exclusion logic but scoped to the selected entity's attributes:

      - Wildcard target format: `ENTITY_CODE.*`, specific target format: `ENTITY_CODE.attrName`
      - When toggling `ENTITY_CODE.*` ON: mark all specific attributes for that entity as pending-delete (if granted) or remove pending-add.
      - When toggling a specific attribute OFF while `ENTITY_CODE.*` is effectively granted: remove wildcard, add pending-add for all OTHER attributes, handle the unticked one.
      - When all specific attributes are toggled ON: auto-promote to `ENTITY_CODE.*`.
      - Use `this.selectedEntity!.attributes` to enumerate all specific attributes in the group.

2. Add helper methods:
   - `private getAllEntityCodes(): string[]` - returns `this.catalogEntries.map(e => e.code)`
   - `private getEntityAttributes(entityCode: string): string[]` - finds the catalog entry by code and returns its attributes array
   - `private checkAutoPromoteEntity(action: string): void` - checks if all entities are effectively granted and auto-promotes
   - `private checkAutoPromoteAttribute(entityCode: string, action: string): void` - same for attributes
