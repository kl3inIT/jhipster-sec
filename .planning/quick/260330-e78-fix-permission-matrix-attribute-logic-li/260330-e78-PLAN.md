<objective>
Fix the permission matrix component to implement three Jmix-style attribute permission behaviors:

1. **Wildcard attribute persistence** -- The '*' wildcard row already exists in the attribute table and saves `entityCode.*` as the target. Verify this works correctly (it appears to already work based on code analysis -- `buildAttributeRows` prepends the wildcard row and `togglePermission` sends `row.target` which is `entityCode.*`).

2. **Modify-implies-view** -- When an attribute has EDIT (modify) permission granted (either explicitly or via wildcard), the VIEW checkbox should be visually checked and disabled. This mirrors Jmix behavior where modify access inherently grants view access.

3. **Entity-level wildcard (*)** -- Add a wildcard row `*` to the entity permissions table. When ticked for a CRUD action, it means "grant this action for ALL entities." Individual entity checkboxes should be disabled when the wildcard is active for that action.

Purpose: Align permission matrix UX with Jmix-style permission semantics so admins get correct visual feedback about implied permissions.
Output: Updated permission matrix component with all three behaviors and passing tests.
</objective>
